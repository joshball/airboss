# Scheduler

Local launchd-driven scheduler installed by the [`scheduled-jobs`](~/src/_me/ai/agent-skills/skills/scheduled-jobs/SKILL.md) skill.

Source skill: `~/src/_me/ai/agent-skills/skills/scheduled-jobs/`.

## First-time setup (per machine)

`.scheduler-config` is gitignored because it contains machine-local paths. On a fresh checkout, regenerate it by running the bootstrap from the skill checkout:

```bash
~/src/_me/ai/agent-skills/skills/scheduled-jobs/installer/install-into-repo.sh
```

This re-runs the interactive prompts, regenerates `.scheduler-config`, and refreshes the manager scripts in place. Idempotent.

After bootstrap, register all enabled jobs:

```bash
bun run schedule install   # or: scripts/scheduler/install.sh
```

## Commands

| Command                    | What                                                         |
| -------------------------- | ------------------------------------------------------------ |
| `./list.sh`                | List installed jobs + schedule + last run + last status      |
| `./new-job.sh <name>`      | Scaffold a new job in `../scheduled-jobs/<name>/`            |
| `./install.sh`             | Generate plists for every enabled job; `launchctl load` them |
| `./uninstall.sh`           | Unload + remove plists for this repo                         |
| `./uninstall.sh --all`     | Remove plists for every repo (global cleanup; prompts)       |
| `./uninstall.sh --dry-run` | Show what would be removed; combine with `--all`             |
| `./status.sh <name>`       | Tail recent log for one job (`--err`, `--runs`, `--lines=N`) |

If `package.json` was wired up at install time, equivalent commands are available as `bun run schedule`, `bun run schedule install`, `bun run schedule uninstall`, `bun run schedule status`, `bun run schedule new`.

## Files

- `.scheduler-config` -- generated by the bootstrap; gitignored. Sourced by every manager script.
- `lib/common.sh` -- shared bash helpers (toml parsing, plist generation, locking)
- `templates/` -- copied into new jobs by `new-job.sh`
- `.cache/` -- run state, locks (gitignored)

## Job authoring

Jobs live in `../scheduled-jobs/<name>/` (one directory per job). Each job has:

- `job.toml` -- declarative cadence + behaviour
- `run.sh` -- the actual work (kept tiny; the wrapper handles PR mechanics)
- `README.md` -- what + why

See [`SKILL.md`](~/src/_me/ai/agent-skills/skills/scheduled-jobs/SKILL.md) for the full schema and authoring guide.

## Plist namespace

`com.scheduled-jobs.<repo-slug>.<job-name>.plist` (so `launchctl list | grep scheduled-jobs` is legible).

Logs land in the directory recorded in `.scheduler-config` -- typically `~/Library/Logs/scheduled-jobs/<repo-slug>/<job-name>/`.

## Worktree isolation

Every job runs in an isolated git worktree under `../<scheduled-jobs-worktrees>/<slug>/<job>-<stamp>/`, branched from `base_branch` (default `main`). The user's main checkout is never touched. The worktree is torn down on exit; the branch is deleted unless a PR was successfully opened.
