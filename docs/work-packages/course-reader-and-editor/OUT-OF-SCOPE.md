## WX chart embedding (reconciled 2026-05-17)

Status: Shipped (component) / Follow-on WP (directive parser)

Reconciliation note:
The original scope deferred chart rendering and shipped `<CourseStepChart>` as a placeholder stub. The component shipped as a REAL chart renderer instead: it fetches an SVG from the study app's `/api/charts/[...slug]/chart.svg` catch-all endpoint (`apps/study/src/routes/api/charts/[...slug]/+server.ts`), which reads `data/charts/wx/<slug>/...` from disk. In development the slug shows as a `<figcaption>`; in production only the chart renders. This OUT-OF-SCOPE entry, the spec, and the test plan originally described a stub that no longer exists -- they were reconciled in the 2026-05-17 review-fixes PR.

What remains deferred:
Only the `:::chart` markdown-directive parser (its own row in the summary table above). `<CourseStepChart>` is a real component, but no content render path mounts it yet -- node bodies and step bodies are Markdown, and the directive parser that would detect `:::chart slug="..."` blocks and mount the component is the gating concern. Until that parser ships, the component is reachable only by its own unit test.

Trigger that fires the follow-on:
The `:::chart` embed syntax stabilizes and the project's Markdown processor gains directive support.

## Truth-aware wx engine integration

Status: Follow-on WP

What was postponed:
The truth-aware generator engine that authors weather scenarios with known truth (per the user's saved memory `project_truth_aware_generators.md`). The engine produces commentary across 4 layers and 3 anchoring stages. None of that surfaces in this WP.

Why:
The engine is its own pedagogy primitive, separate from course rendering. A course can reference an engine output (e.g., a step body says "decode the METAR generated by the wx engine on 2026-05-09"), but the engine itself is a separate surface with its own scope.

Trigger that fires the follow-on:
The wx engine ships AND a course step needs to embed an engine-authored scenario. Likely shape: a `<CourseStepScenario id="..." />` component analogous to the chart stub, with the engine surfacing scenarios via a stable id.

Implementation pattern when triggered:
Mirror the chart stub: a small Svelte component that takes a stable id, looks up the engine output, renders it. The engine WP ships the data + the component; this WP's step reader already has the embedding hook.

References:

- Project memory: `project_truth_aware_generators.md`
- Project memory: `project_encoded_text_family.md`

## Personal-course authoring UI

Status: Deferred

What was deferred:
A study-app surface that lets a learner author their own personal course (per the course-primitive WP's deferred personal-course feature). `COURSE_KINDS.PERSONAL` is reserved as a value, but neither this WP nor the foundation WP authors a UI for it.

Why:
Per [LEARNING_PHILOSOPHY.md](../../platform/LEARNING_PHILOSOPHY.md) principle 11 and [course-primitive/OUT-OF-SCOPE.md](../course-primitive/OUT-OF-SCOPE.md): the chef and the diner are different people. Personal-course authoring crosses the line.

Trigger to revisit:
Mirrors the course-primitive WP's same-named entry. When the user (Joshua, or another learner) explicitly asks for "a course I author for myself" AND the goal + ad-hoc nodes primitive is not enough.

Implementation pattern when triggered:
Add a study-side authoring surface (NOT in hangar -- this is learner-authored content). Personal courses are user_id-scoped; the seed pipeline does not touch them; reads/writes go through the BC directly. The reader pages built in this WP can be reused for the read path; the editor pages built in this WP cannot (they assume YAML-as-source).

References:

- [course-primitive/OUT-OF-SCOPE.md](../course-primitive/OUT-OF-SCOPE.md) -- Personal-course authoring UI

## Cross-course mastery dashboards

Status: Deferred

What was deferred:
A surface that aggregates mastery across multiple courses in a goal. The detail page renders one course at a time; the goal-detail page lists courses but does not roll mastery up across them.

Why:
Per the prompt that opened this WP: cross-course rollup is its own WP. With one authored course, cross-course rollup degenerates to per-course rollup; the value of a dedicated dashboard surfaces only when the user holds multiple courses.

Trigger to revisit:
The user holds at least 2 active courses in one goal AND has expressed a need to see "across both, how am I doing?"

Implementation pattern when triggered:
Likely a new lens (or a wrapper around `getGoalNodeUnion`'s union) that aggregates mastery per knowledge node across all goal_course rows. Surface as a panel on `/program/goals/[id]` or as its own route.

References:

- WP prompt (2026-05-09): "Cross-course mastery dashboards (one course at a time; cross-course rollup is its own WP)"

## Hangar editor for syllabus YAML

Status: Rejected

What was rejected:
A hangar-app editor for syllabus YAML files (PPL ACS, IR ACS, etc.).

Why:
Per the prompt that opened this WP: syllabi are authored hand-in-YAML. Syllabus content is regulatory (ACS / PTS / 61.31), edition-locked, and changes only when the FAA publishes a new edition (which is rare and high-stakes). A hangar editor for syllabi would make casual editing too easy and undermine the edition-lock guarantee from [ADR 016](../../decisions/016-cert-syllabus-goal-model/decision.md).

Course YAML is the right authoring target for a hangar editor because courses are mutable by design and instructor-pedagogy-shaped. Syllabi are a different kind of authoring with a different cadence and a different review surface (transcribe-once, version-on-publish).

A re-decision would have to clear: a syllabus authoring cadence that justifies in-app editing (e.g., school-syllabus authoring -- but `syllabus.kind='school'` is itself deferred and zero rows exist).

References:

- WP prompt (2026-05-09): "Hangar editor for syllabus YAML"
- [ADR 016](../../decisions/016-cert-syllabus-goal-model/decision.md) syllabus authoring section

## Multi-syllabus overlay picker

Status: Deferred

What was deferred:
A picker on the course-detail page that lets the learner choose which syllabus to overlay against (today the page picks the highest-weight `goal_syllabus` deterministically).

Why:
With one authored course and a learner pursuing one syllabus (PPL), the picker is theoretical. Adding it now is speculative UX.

Trigger to revisit:
The user holds 2+ syllabi in one goal AND wants to see the same course's coverage against each. Likely scenario: pursuing PPL + IR concurrently, taking a weather course, wanting to see "this course covers X% of PPL ACS Area I.C and Y% of IR ACS Area III.A."

Implementation pattern when triggered:
A small dropdown above the cert-gaps panel. Picker options come from `goal_syllabus` rows of the active goal. Selection re-runs `courseWithCertOverlayLens` with the new `syllabusId`. URL state via `?cert=<syllabus_id>` so deep-links work.

References:

- spec.md "Cert overlay: which syllabus" section

## Multi-author concurrency on the same course

Status: Deferred

What was deferred:
File-locking, conflict detection, or last-writer-wins UX for the hangar editor.

Why:
The hangar app is single-user today (Joshua). Adding concurrency primitives now is speculative.

Trigger to revisit:
A second author starts editing courses (or even better: two people are simultaneously authoring different sections of the same course).

Implementation pattern when triggered:
Either an advisory lock surfaced in the UI (a banner that says "Bob is editing this course; your edits may conflict") OR an optimistic-concurrency check on save (the form carries a "version" token; the save action verifies the file's mtime matches before overwriting).

## Encoded-text per-tab content

Status: Deferred

What was deferred:
Per-tab content for the Decode / Understand / Triage strip. Today the tabs are visual hints only; clicking them does nothing (or scrolls to a section anchor). A real per-tab implementation would filter the node body, surface tab-specific cards, or show tab-specific scenarios.

Why:
The tab strip's purpose in this WP is orientation -- "this content is part of a skill ladder." Surfacing actual tab-specific content requires the encoded-text content itself to be authored against the skill-ladder shape, which is not yet done.

Trigger to revisit:
The encoded-text family content (METAR/TAF/PIREP/AIRMET/SIGMET nodes) has authored skill-ladder cards or scenario sets, AND the user (or another learner) explicitly asks "show me only the Decode content."

Implementation pattern when triggered:
A small lens-style filter applied to the node's cards / scenarios. Tab state via URL `?ladder=decode|understand|triage`. The tab strip becomes interactive; the page filters the practice surfaces to the chosen ladder rung.

References:

- Project memory: `project_encoded_text_family.md`
- spec.md "Encoded-text family rendering" section

## Course content authoring (the actual weather course)

Status: Follow-on WP

What was postponed:
Authoring the actual weather-comprehensive course (or whatever the first instructor course is named). This WP ships the editor; a separate WP authors the content using the editor.

Why:
Authoring content and authoring the surface are different concerns with different review criteria. The surface review checks "does the editor work?"; the content review checks "is the pedagogy good?". Bundling them risks signing off on the surface based on the content quality (or vice versa).

Trigger that fires the follow-on:
This WP ships, the editor is reviewed, and Joshua (or another instructor) starts authoring real course content.

Implementation pattern when triggered:
Use the editor. The follow-on WP's deliverable is YAML files + node body fills, not code.

References:

- WP prompt (2026-05-09): "the first real course (weather-comprehensive or similar) will be authored AFTER this WP ships, against the new editor"

## `:::chart` markdown extension parser

Status: Follow-on WP

What was postponed:
A markdown extension that parses `:::chart slug="..."\n:::` blocks in step body_md or node body_md and mounts the `<CourseStepChart>` component. This WP ships the component; the markdown integration is its own scope.

Why:
The chart component is a Svelte primitive; the markdown integration depends on which markdown processor the project uses (the existing 7-phase node renderer). Adding a directive parser is a project-wide change that affects every markdown-rendered surface, not just the course step reader.

Trigger that fires the follow-on:
The syntax for embedding stabilizes. Real chart rendering already shipped (`<CourseStepChart>` fetches a real SVG -- see the reconciliation note above), so the directive parser is now the only gating concern. Today an author can embed the component manually only by authoring a Svelte file, which they do not -- node bodies and step bodies are Markdown.

Implementation pattern when triggered:
Mirror the existing markdown extensions (callouts, fenced code blocks). Hook into the markdown processor. Test against the existing renderer surfaces.

References:

- spec.md "Step reader" section

## In-place reorder via drag-and-drop

Status: Deferred

What was deferred:
Drag-and-drop reordering for sections (in the course editor) and steps (in the section editor). This WP ships ordinal-edit-via-number-input only.

Why:
Drag-and-drop is a real UX win, but it's not free: it needs a sortable component, touch-device support, accessibility (keyboard reorder), and a visual indicator. None of that is critical to "the editor works at all." A keyboard-only ordinal edit is sufficient for single-user authoring at this scale.

Trigger to revisit:
The author (Joshua) reports that ordinal-edit-by-number is creating real friction (e.g., reordering 20+ steps becomes painful). At that point a sortable library lands.

Implementation pattern when triggered:
Pick a Svelte-compatible sortable library (svelte-dnd-action is the common choice). Add ARIA support for keyboard reorder. The save action remains "re-emit YAML with new ordinals + re-seed."

References:

- spec.md hangar editor sections
