# Out of Scope: Source cache flat naming

Deferred items, why they're deferred, and the trigger that should make
us revisit each. Future agents and humans: do not build these without
the documented trigger. If you think the trigger is hit, surface it
for a decision rather than building silently.

## Summary

| Item                                                  | Status   | Trigger to revisit                                                   |
| ----------------------------------------------------- | -------- | -------------------------------------------------------------------- |
| `.archive/` cleanup of old `source.pdf` references    | Rejected | Never -- frozen artifacts under `docs/.archive/`                     |
| Inline derivative tier layout changes                 | Rejected | Never -- this WP only touches the developer-local cache              |
| DB schema changes for renamed cache                   | Rejected | Never -- the cache rename is invisible to the DB tier                |
| Generated derivatives outside cache (`prompts-out/`)  | Rejected | Never -- those trees follow their own naming, not part of this WP    |

## `.archive/` cleanup of old `source.pdf` references

Status: Rejected

What was rejected:
Updating references to `source.pdf` / `source.xml` / `_errata/` inside `docs/.archive/`.

Why:
Pre-pivot or otherwise archived references are frozen artifacts; rewriting them would falsify history. The archive's purpose is preservation, and the project rule says "Archive superseded docs, never delete." That rule extends to rewriting too -- archived content describes the world as it was, not as it is.

References:

- [spec.md](./spec.md) -- "Out of Scope (explicit)" listed `.archive/` cleanups
- airboss/CLAUDE.md -- "Archive superseded docs, never delete. Preserve directory structure in `.archive/`."

## Inline derivative tier layout changes

Status: Rejected

What was rejected:
Reorganizing the committed `handbooks/`, `ac/`, `acs/`, `aim/`, `regulations/` derivative trees (markdown, figure PNGs, table HTML, derivative manifests) under the repo root.

Why:
This WP is scoped to the developer-local cache tier (gitignored) only. The inline derivative tier is committed content with stable consumer paths used by every downstream BC. Touching it would multiply the surface area and risk; a separate WP (if ever needed) would own that.

References:

- [spec.md](./spec.md) -- "Out of Scope (explicit)" stated only the cache tier is touched
- [spec.md](./spec.md) §"Cache-tier vs inline-derivative-tier (important distinction)"
- [ADR 018](../../decisions/018-source-artifact-storage-policy/decision.md) -- three-tier policy still stands post-supersedure of filenames

## DB schema changes for renamed cache

Status: Rejected

What was rejected:
Updating `handbook_section`, `reference`, or any related table to record the new cache paths.

Why:
The cache rename is invisible to the DB tier. Derivative manifests record `source_url` and `source_pdf_sha256` (FAA URL + content hash) rather than cache paths, so DB rows reference URLs and hashes not filenames. The DB tier sees no path change.

References:

- [spec.md](./spec.md) -- "Out of Scope (explicit)" stated no DB schema changes
- [spec.md](./spec.md) §I risk register row on inline derivative manifests

## Generated derivatives outside cache (`prompts-out/`)

Status: Rejected

What was rejected:
Rewriting paths or rules for trees generated by tooling outside the cache (e.g. `tools/handbook-ingest/prompts-out/`).

Why:
Those trees follow their own naming and lifecycle. Verified at WP time: `rg -l 'source\.pdf|_errata' tools/handbook-ingest/prompts-out/` returned zero hits as of 2026-04-29; prompts-out contains extraction output (chapter markdown), not cache paths. No real coupling exists to break.

Trigger to revisit:
If a future change ever bakes cache paths into prompts-out artifacts, that's a separate bug to fix when it surfaces -- not the same effort as cache-tier rename.

References:

- [spec.md](./spec.md) -- "Out of Scope (explicit)" pointed at this
- [spec.md](./spec.md) §I risk register "Decision: dropped" on prompts-out

## Multi-edition coexistence policy

Status: Deferred

What was deferred:
A formalized "two editions of the same handbook side by side in the cache" policy. The cache holds one edition per slug today; handbook slugs include the edition letter, so a new edition would land as a flat sibling.

Why:
ADR 020 owns handbook edition and amendment policy. A multi-edition migration policy is content-policy work, not cache-layout work, and stacking it into the rename WP would have inflated scope past one reviewable unit.

Trigger to revisit:
When FAA publishes a new edition of an already-cached handbook AND the consumer needs to read both side by side (today's slug-bump pattern is sufficient for "swap to the new one").

Implementation pattern when triggered:
Extend [ADR 020](../../decisions/020-handbook-edition-and-amendment-policy.md) with the coexistence rules; the cache layout already supports flat siblings without changes.

References:

- [spec.md](./spec.md) -- "Out of Scope (explicit)" punted multi-edition policy to ADR 020
- [ADR 020](../../decisions/020-handbook-edition-and-amendment-policy.md) -- handbook edition and amendment policy

## `bun run sources verify-cache` post-migration command

Status: Deferred

What was deferred:
A `bun run sources verify-cache` command that walks the cache, cross-checks every manifest entry against on-disk files, and reports missing artifacts.

Why:
The post-merge findings document recommends this for **future** migrations after PHAK MOSAIC errata was silently lost during the one-shot migration. The current cache state is consistent (PHAK was self-healed via re-apply), so the command isn't needed today. Baking the verifier into the next migration script is the right time, not retrofitting it for a cache that's already stable.

Trigger to revisit:
At the next cache migration of any kind (filename rewrite, layout shift, schema bump). Author the verifier as part of that migration's script before any rename ops run.

Implementation pattern when triggered:
Walk `<cache>/<corpus>/manifest.json` for each corpus, for each entry assert the `filename` exists on disk, fail loud and exit non-zero on any miss. Mirror the post-migration verification recommended in [post-merge-findings.md](./post-merge-findings.md) "Recommended actions" §1.

References:

- [post-merge-findings.md](./post-merge-findings.md) -- "Recommended actions" §1 and §3 detail the verifier shape
- [post-merge-findings.md](./post-merge-findings.md) -- the PHAK MOSAIC errata-loss timeline that motivated it

## Migration script preservation in `docs/.archive/`

Status: Deferred

What was deferred:
A revised policy where migration scripts get archived to `docs/.archive/migrations/<date>-<name>.ts` rather than deleted in a same-PR cleanup commit. The spec required two-commit lifecycle (commit A adds + runs, commit B deletes) for this WP; the post-merge findings argue checked-in archived scripts would have made the PHAK errata loss diagnosable.

Why:
Changing the lifecycle policy mid-WP would have collapsed two unrelated decisions into one PR (cache rename + script-preservation policy). The right time to revisit is at the next migration when the policy can be written from the start.

Trigger to revisit:
At the next cache migration of any kind, before the migration script gets authored. Choose between "delete in same PR" (current rule) or "preserve in `docs/.archive/migrations/`" (post-merge recommendation) at that time.

Implementation pattern when triggered:
Author the migration script at `docs/.archive/migrations/<YYYY-MM-DD>-<slug>.ts` from the start, or move it there in the cleanup commit instead of deleting. Preserves traceability without leaving runnable code on main.

References:

- [post-merge-findings.md](./post-merge-findings.md) -- "Recommended actions" §2 proposes the archive policy
- [spec.md](./spec.md) §G "The script lifecycle is two adjacent commits" -- current policy
