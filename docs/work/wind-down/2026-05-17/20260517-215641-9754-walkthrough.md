# Walkthrough -- project-wide 10x review session

This session was a code-review-and-fix effort across the whole project (~393k LOC).
No new user-facing features were built. The fixes are correctness, security, a11y,
and type-quality changes spread across every surface. There is no single new page
to demo; instead, the items below are the user-observable behaviors that changed
and are worth a manual spot-check.

## Weather decoding (apps/study, apps/hangar -- wx surfaces)

**Where**: any surface that renders a decoded METAR/TAF (the wx drill/practice pages
under the study app, and the wx-catalog views in the hangar app).

**How to test manually**:

1. Find or author a METAR containing `M1/4SM` visibility.
2. Verify the decoded visibility now reads `1/4 SM` (the threshold) with a
   "less than" qualifier in the prose -- previously it fabricated `1/8 SM`.
3. Find a METAR with a moderate thunderstorm (`TSRA`, not just heavy `+TSRA`).
4. Verify the convective "why" explanation now renders for it.
5. Find a METAR reporting a `CB` or `TCU` cloud layer.
6. Verify the "CB layer marks the convective cell" annotation now appears.

**What to look for**: decoded weather values match the raw observation exactly;
no invented numbers.

## UI components -- duplicate-instance accessibility

**Where**: any page that renders two of the same component with the same label
(e.g. two "Include archived" checkboxes, two same-titled panels, the home page's
repeated `goal` info-tips).

**How to test manually**:

1. Open a page with a repeated `Checkbox` / `FormField` / `InfoTip` /
   `MetricExplainer` / `PanelShell`.
2. Click the second instance's `<label>` -- it should toggle/focus its OWN
   control, not the first one.
3. With a screen reader, trigger validation on the second `FormField` -- the
   announced error must be that field's, not the other's.

**What to look for**: label-to-control association and `aria-describedby` resolve
per-instance; no duplicate DOM ids.

## Markdown links with query strings

**Where**: anywhere `renderMarkdown` output is shown (help drawer content,
rendered reference sections, any markdown-authored copy).

**How to test manually**:

1. Find rendered markdown containing a link with a query string, e.g.
   `[chart](https://example.com/x?a=1&b=2)`.
2. Click it; inspect the resulting URL.

**What to look for**: the `href` is `...?a=1&b=2`, not the previously-corrupted
`...?a=1&amp;b=2`.

## Sim replay debrief

**Where**: the sim app debrief/scrubber after running a scenario with a fault.

**How to test manually**:

1. Run a sim scenario where a fault triggers mid-run (not on a snapshot tick).
2. Open the debrief replay scrubber.
3. Verify the timeline marks "fault fired here" at the correct tick.

**What to look for**: fault-edge markers appear on the replay tape for faults
that fire on any tick, not only snapshot ticks.

## Known caveats

- Several test suites show pre-existing failures when run under `bun test`
  (Vitest-API tests using `vi.hoisted` / `vi.setConfig`, and a transpiler
  `Cannot access ... before initialization` issue). These are environmental,
  not regressions from this session -- verified identical on a clean `origin/main`.
- DB-integration tests require a live PostgreSQL + `DATABASE_URL`; they fail in
  worktrees without `.env`. Not a regression.
