# wp-hangar-registry Phases 4-5

Builds on the Phases 1-3 foundation merged in PR #92 (TOML codec, DB schemas, generic job queue).

## What shipped in this PR

### `libs/hangar-sync/` (new lib, `@ab/hangar-sync`)

- `paths.ts` -- absolute + relative paths for the three files sync writes; advisory-lock key constant.
- `to-domain.ts` -- `hangarReference` / `hangarSource` row -> domain `Reference` / `Source` converters.
- `emit-toml.ts` -- wraps `encodeReferences` / `encodeSources` to produce the two TOML bodies keyed by absolute path.
- `emit-aviation-ts.ts` -- regenerates `libs/aviation/src/references/aviation.ts` per Option 3 (keep the TS artifact so study's client bundle stays fast). Deterministic, header-warns "do not edit by hand", sorts by id.
- `detect-drift.ts` -- compares DB-derived TOML against on-disk bodies; returns per-row drift entries + the set of files that need a rewrite.
- `detect-conflict.ts` -- compares current DB revs against the most recent successful sync's `rev_snapshot`; any id whose rev advanced past the baseline is a conflict.
- `git.ts` -- `ProcessRunner` interface backed by `node:child_process.spawn`. Helpers for `git add`, `git commit`, `git push`, `git checkout -b`, `gh pr create` (body piped via stdin).
- `commit-and-maybe-pr.ts` -- two-mode runner (`commit-local` default, `pr` prod). Writes files, stages by name, commits, optionally pushes + opens a PR.
- `run-sync-job.ts` -- the `sync-to-disk` job handler. `executeSync` is the pure state machine; `runSync` wraps it with the DB transaction + pg_advisory_xact_lock + real writers; `runSyncJob` wraps `runSync` with the `@ab/hangar-jobs` `JobContext` adapter.
- `index.ts` -- public exports.

### Schema extension

- `libs/db/src/hangar.ts` -- added `rev_snapshot: jsonb | null` column to `hangar.sync_log`. Persists per-id revs that the successful sync committed, so the next sync's conflict detector has a baseline.
- `drizzle/0002_hangar_sync_rev_snapshot.sql` + snapshot JSON + journal entry.

### Constants

- `SYNC_OUTCOMES.NOOP` added to `libs/constants/src/jobs.ts` (sync ran but DB matched disk).

### Worker boot (Phase 5)

- `apps/hangar/src/lib/server/jobs.ts` -- `JobHandlers` registry mapping `JOB_KINDS.SYNC_TO_DISK -> runSyncJob`. Ready for WP3 to add fetch/upload/extract/build/diff handlers without touching the worker wiring.
- `apps/hangar/src/hooks.server.ts` -- boots one worker per module load: `recoverOrphanedRunning()` then `startWorker({ handlers })`. Error-swallowing so a DB hiccup at boot doesn't take the app down. `beforeExit` requests a graceful drain.
- `apps/hangar/svelte.config.js` -- alias entries added for `@ab/aviation`, `@ab/hangar-jobs`, `@ab/hangar-sync`.

### Minor

- `libs/aviation/src/index.ts` -- re-exports `encodeReferences` / `encodeSources` / `decodeReferences` / `decodeSources` (needed by the sync service + its tests).
- `libs/aviation/src/toml-codec.ts` -- four `as Record<string, unknown>` casts converted to `as unknown as Record<...>` so svelte-check in study is happy once the codec is reachable through the public export.
- `vitest.config.ts` -- aliases for `@ab/audit`, `@ab/hangar-jobs`, `@ab/hangar-sync`.
- `.env` copied into the worktree (gitignored; needed for vitest setup to load DATABASE_URL so drizzle's connection module imports cleanly).
