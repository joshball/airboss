## Decisions

### HelpSearch: option (a) was already done

`libs/help/src/ui/HelpSearch.svelte` is fully app-agnostic -- it searches
the global `helpRegistry` singleton, which every app's `$lib/help/register`
populates at module-eval. No refactor needed; AppHeader just exposes a
`helpSearch` snippet slot so apps mount it without `@ab/ui` taking a
direct dependency on `@ab/help` (which would form a cycle: `@ab/help`
already imports `@ab/ui` for Drawer + focus-trap).

Per-app context bias (rank in-app pages first when searching from sim) is
deferred: the registry already supports `surface:` filters and the search
input lets users type them. **Trigger to revisit**: when sim or avionics
authors its first real help page set, add `currentApp`/`currentPath`
props on the HelpSearch component and pre-bias bucket-1 ranking by the
caller's app. (At that point, ~5 LOC change in `search-core.ts`.)

### Preferences link

`preferencesHref` prop added but unused -- no app passes one, no
`/preferences` route exists. Preserves the slot without inventing a
non-existent route. **Trigger to revisit**: when the cert-syllabus or
account-settings WP authors a preferences page.

### Cross-origin sign-in redirect

`studyLoginUrl(event)` produces the canonical
`https://study.<root>/login?redirectTo=<currentCrossOriginUrl>`. The
study login form reads `redirectTo` and validates with `isSafeRedirect`
(same-origin only). So a cross-origin redirect today still falls back to
study's home -- the post-login round-trip back to sim/avionics/flightbag
is **not yet honored**. Pre-existing limitation, surfaced now because v2
relies on the redirectTo flow. **Trigger to revisit**: out of scope for
this PR; addressed by ADR 024 (cross-app auth resolution).

### Hangar `/help` route

Hangar already authored help pages but lacked a top-level surface. v2
needed the help link to resolve, so I added `/help` and `/help/[id]`
that mirror study's loaders (just `helpRegistry.getAllPages()` and
`loadById` plus `HelpLayout`). This is the smallest possible surface
for the existing data; expanding it (a hangar-specific landing) is a
follow-on if the user wants it.

### Brand "APP NAME" sizing

Used `var(--type-heading-3-size)` (1.125rem) with
`var(--type-heading-1-weight)` (700) -- the closest existing tokens for a
strong app title that doesn't compete with page H1s. No new tokens
introduced.
