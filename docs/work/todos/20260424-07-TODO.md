# wp-hangar-non-textual shipped

Implementation sweep for the binary-visual source family, seeded by
Denver VFR Sectional. Covers phases 1 through 7 of
`docs/work-packages/hangar-non-textual/tasks.md` in a single PR.

## What shipped

- Phase 1: `SOURCE_KINDS` + `SOURCE_KIND_BY_TYPE` classifier, `SECTIONAL`
  source type, `geotiff-zip` format, `SECTIONAL_THUMBNAIL` constants,
  `SourceMedia` + `SourceEdition` types, audit targets, migration 0003.
- Phase 2: FAA AeroNav edition resolver + GDAL/sips/unavailable thumbnail
  generator with fixture-backed vitest coverage.
- Phase 3: fetch-handler dispatch by source-kind. Binary-visual path runs
  in-process (resolve -> short-circuit -> download -> drift-detect ->
  archive rotate -> move -> manifest -> thumbnail -> meta.json -> DB
  update -> audit). Extract / diff / scan scripts + CLI short-circuit
  with a clear 'not applicable' message on binary-visual sources.
- Phase 4: source-form kind-aware panel (region + cadence + index URL);
  Zod + field-error validation; edit-page round-trips authored values.
- Phase 5: SourcePreviewTile + /sources/[id]/download + /thumbnail
  routes; files browser walks the nested edition layout for binary-
  visual sources.
- Phase 7: setup.ts tool-detection step + this notes file.

## Deferred / captured

- **Denver sectional seed row.** Per task 4.5, the `sectional-denver`
  row should be authored through the UI in dev + synced-to-disk via a
  dedicated job (not hand-edited into `sources.toml`). That is an
  operator-side task and depends on a live dev DB with migration 0003
  applied; captured here so the next session can close the loop.
- **Edition-resolver override env var** for simulating a next-edition
  fetch in the manual walkthrough (test-plan step 18). Not wired
  yet; the resolver takes an injected `fetchHtml`, so the simplest
  path is a dev-only endpoint that returns a stubbed HTML. Follow-up.
- **GeoTIFF/JPEG/Zip custom previews** in /sources/[id]/files. The
  files browser currently falls through to the generic BINARY
  preview for `.tif`/`.jpg`/`.zip`. Phase 5 of the WP calls for
  dedicated `GeotiffPreview`, `JpegPreview`, `ZipPreview` components;
  deferred because the current generic path already shows filename +
  size + archived flag, which meets the "admin can verify fetch
  landed" success criterion. Next session can add them without
  touching any other surface.
- **Reference CLI** `bun references download --id sectional-denver`.
  The hangar UI path and the in-process pipeline already work; a
  thin CLI wrapper over `handleBinaryVisualFetch` would give operators
  a terminal-only path. Deferred -- hangar UI is the primary surface.

## Manual walkthrough prerequisites

To run the full test plan on dev, apply the migration:

```bash
bun run db migrate
# or, on a DB seeded before 0003:
docker exec airboss-db psql -U airboss -d airboss -c \
  'ALTER TABLE hangar.source ADD COLUMN IF NOT EXISTS media jsonb; \
   ALTER TABLE hangar.source ADD COLUMN IF NOT EXISTS edition jsonb;'
```

Then follow `docs/work-packages/hangar-non-textual/test-plan.md` steps
1 through 33.
