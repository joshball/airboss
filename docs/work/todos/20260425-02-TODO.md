### PR #180 -- `fix(memory-browse): review crash, filter alignment, density, facets, paging, grouping`

- `/memory/review` 500 fix. `getDueCards` (and the snooze auto-suspend path) interpolated a JS `Date` directly into a raw drizzle `sql` template; postgres-js sent `Date.toString()` and rejected the param with `ERR_INVALID_ARG_TYPE`. Both call sites switched to `.toISOString()` to match the working pattern in `scenarios.ts`. Verified by running the offending query against the live dev DB before and after.
  - `libs/bc/study/src/cards.ts:291`
  - `libs/bc/study/src/snooze.ts:376`
- Filter row alignment. Status select and Apply/Reset previously sat on a different baseline from Search/Domain/Type/Source; Apply/Reset hung below the filter card edge. Unified all blocks under a phantom-label spacer.
- Card density. Replaced the vertical `<dl>` stat list (10 visible rows per card) with an inline `State / Due / Stab / Last` strip. Cards are now ~3 rows.
- Page-size picker. `10 / 25 / 50 / 100` (default 25). Lives in `routes.ts` as `QUERY_PARAMS.PAGE_SIZE`; values clamped to `BROWSE_PAGE_SIZE_VALUES`.
- Group-by picker. `none / domain / type / source / status / state`. Server returns a flat list; client buckets and sorts by group label.
- Filter facet counts. New `getCardsFacetCounts` BC function returns `(N)` hints next to each filter option, computed against the *other* active filters so the numbers always answer "if I pick this, what would I see?"
- Subhead copy clarified from "Your deck of memory items" to explicit framing: every card on the user's account, authored + course-seeded.

### PR #182 -- `refactor(ui): extract FilterCard, FilterChips, Pager, ResultSummary`

Same filter+chips+pager pattern was open-coded in `/memory/browse`, `/reps/browse`, and `/knowledge/`, with subtly drifted CSS and chip-href logic. Extracted four shared components into `@ab/ui` and rewrote all three pages to consume them.

- `libs/ui/src/components/FilterCard.svelte` -- rounded box + auto-fit grid for filter selects + Apply/Reset row. Accepts a `controls` snippet for the actual `<select>`s/`<input>`s and an optional `hidden` snippet for hidden form inputs.
- `libs/ui/src/components/FilterChips.svelte` -- active-filter chips + Clear all link. Takes a typed `FilterChipDef[]` with **precomputed `removeHref`** so pages don't rebuild URLs in the template.
- `libs/ui/src/components/Pager.svelte` -- Previous / Page N of M / Next.
- `libs/ui/src/components/ResultSummary.svelte` -- "Showing X-Y of Z noun matching your filters" line. Pluralizes via `noun`/`nounPlural`.

Per-route list/card rendering left in place because the contents diverge meaningfully (memory has stats + group headings + restore form; reps has option count + difficulty colors; knowledge has mastery bar + domain section headings). A `BrowseList` HOC would have one prop per variant, worse than three small templates that say what they mean.

Net `+614 / -687` (-73 lines). No visual change.

### PR #18x -- `refactor(browse): list-item base + reps/knowledge feature parity`

User read PR #182 and called out that the *list items themselves* should also be a base component, and that reps and knowledge should match memory's feature surface (search, page-size, group-by, facet counts, chips). Both shipped.

- New `BrowseList.svelte` -- `<ul>` shell with group-heading + just-created highlight rendering. Groups + item snippet pattern.
- New `BrowseListItem.svelte` -- the rounded card shell with `title`/`meta`/`stats`/`extra`/`trailing` snippets. The three pages override only the snippet contents.
- New `BrowseViewControls.svelte` -- the "Group by" + "Per page" selectors (was inline duplication on memory-browse).
- New `getScenariosFacetCounts` BC + `search` field on `ScenarioFilters`. Facets exclude their own filter so the `(N)` counts answer "if I pick this option (with my other filters), what would I see?"
- New `listNodesWithFacets` BC for knowledge -- materializes once, derives facets in JS (small table).
- Reps server: page-size, group-by, search, facet pass added; reps-specific group-by enum (`domain`/`difficulty`/`phaseOfFlight`/`source`/`status`/`none`) since cards' `state` doesn't apply.
- Knowledge server: page-size, group-by (defaulting to `domain`), facet pass added. New chips for active filters; default-no-grouping path; `(none)` bucket for nodes without certs/priorities.
- All three pages wired onto the new components. Heavy CSS removed from each route (now owned by the components).
