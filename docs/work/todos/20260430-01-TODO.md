# 2026-04-30 -- Encoded-text family + Weather Scenario Engine shaping

## Context

Captured a large new design direction for the platform: the family of dense, encoded, time-pressured formats pilots read (NOTAMs, METARs, TAFs, PIREPs, AIRMETs/SIGMETs, charts, ATIS, clearance shorthand, etc.). All members share the same skill ladder (decode → understand → triage). Weather is the richest place to start because the pedagogy benefits most from synthetic generation with truth-aware commentary.

Did **no** implementation work this session. Captured shape only -- principles, ideas, course skeletons, a vision doc.

## Open -- blocking acronym verification

Lessons cannot be authored in `course/notams/` until these acronyms are verified against authoritative FAA sources (FAA Order JO 7930.2, AIM 5-1-3, faa.gov NOTAM Search docs, AIS public pages, Notices to Airmen Publication):

- **USNS** -- legacy text-based NOTAM system (verify exact name, dates, what replaced it)
- **FNS** -- Federal NOTAM System (verify; distinct from NMS or same?)
- **NMS** -- NOTAM Manager System (verify name, relationship to FNS)
- **AIDAP** -- distribution protocol (verify full name, role)
- **XMLQ** -- legacy distribution format (verify name, role, retirement)
- **BHIV** -- entry-side system, "behave" in old controller slang (verify name, status, slang authenticity)
- **FICON** -- runway condition reporting (verify current spec, effective dates, RwyCC scale)

Tracked in [course/notams/TASKS.md](../../../course/notams/TASKS.md). Do not author NOTAM lesson content with these unverified.
