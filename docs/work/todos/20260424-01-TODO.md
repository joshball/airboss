## What still needs to happen (Phases 3-10 of the WP)

### TOML-read strategy -- DECIDED (Option 3)

The spec said delete `libs/aviation/src/references/aviation.ts` and rewrite `libs/aviation/src/registry.ts` to parse TOML at boot. But `AVIATION_REFERENCES` is imported by `+page.ts` universal loads in study (runs on both server and client after hydration), and TOML cannot be read via `fs` in the browser.

**Decision (2026-04-24, user):** Option 3. Keep `aviation.ts` as a generated TS artifact; authoritative edits flow TOML -> hangar sync -> regenerate `aviation.ts`. Rationale: TOML is a dev-loop convenience that will be replaced by a DB in production; Option 3 costs nothing now and preserves the consumer contract so the production transition is a one-file swap (regenerate from DB instead of from TOML). Options 1 and 2 were rejected: Option 1 regresses client-nav perf, Option 2 adds Vite-plugin complexity for scaffolding that will be removed.

### Phase 3 -- `@ab/hangar-jobs` lib

Enqueue + worker loop + per-target serialisation + recovery. No runners beyond `sync-to-disk` land here.

### Phase 4 -- `@ab/hangar-sync` lib
