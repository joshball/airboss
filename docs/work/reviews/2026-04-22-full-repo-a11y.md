## Summary

`libs/ui` primitives are on the right track — real `<button>`/`<a>`, real `<label for>`, `aria-invalid`/`aria-describedby` wired on form fields, a focus-trapped `ConfirmAction`, a route-aware skip link, and a live-region on the session page. The problems are concentrated in a few propagatable patterns: a11y regressions in `apps/study/src/routes/(app)/memory/new/+page.svelte` where the page hand-rolled its own form controls and bypassed `TextField`/`Select`; two sim overlays (`ResetConfirm`, `KeybindingsHelp`) that are dialogs in name only (no focus trap, no initial focus, no `Escape` handler, no focus restoration); and focus-visible styling that relies on a translucent `box-shadow` focus ring (≈ rgba `0.35` alpha) which does not meet 3:1 non-text contrast on white in either theme. Heading hierarchy is mostly clean. Color-only signalling shows up in `review` rating buttons (rating-1…4 are distinguished from each other only by `color` on the label) and in a few `role="status"` banners that lack icons or text prefix.

## Propagatable Patterns (top priority)

1. **Focus ring relies on translucent box-shadow only** — `libs/themes/tokens.css:120-122`, inherited by `Button`, `TextField`, `Select`, `StatTile`, `ConfidenceSlider`, `ConfirmAction`, `KbdHint`, `HelpSearch`. `--ab-color-focus-ring: rgba(37, 99, 235, 0.35)` on white surfaces renders well below WCAG 2.4.11 / 1.4.11 3:1 contrast. Every primitive sets `outline: none` and replaces it with this ring — fix the token once and every primitive benefits; ship a separate solid-color fallback ring for high-contrast / forced-colors.
2. **Primitives are silently bypassed** — `apps/study/src/routes/(app)/memory/new/+page.svelte:106-173`, `apps/study/src/routes/(app)/memory/browse/+page.svelte:149-188`, `apps/study/src/routes/(app)/plans/new/+page.svelte`, `apps/study/src/routes/(app)/reps/new/+page.svelte` all re-implement `TextField`/`Select`/`Button`. Every a11y fix we ship in `libs/ui/src/components/TextField.svelte` is invisible to these pages. Root cause: no lint / review gate on raw `<input>` / `<select>` / `.btn` in app code. Convergent fix: migrate every call site to primitives (or at minimum wire the same `id`+`for`, `aria-describedby`, `aria-invalid` contract).
3. **`ConfirmAction` focus trap is shallow** — `libs/ui/src/components/ConfirmAction.svelte:80-101`. Tab is trapped between `confirmEl` and `cancelEl` only while they are the event target; when the form has hidden inputs or the Confirm button is inside a `<form>`, Tab from outside the panel (shifted from before the trigger) lands on the form before Confirm. Also Escape only cancels when keydown originates inside the panel — click-out is not supported at all and the panel has no `aria-describedby` for context. Since this widget is the canonical destructive confirmation across the app, leaks cascade.
4. **Dialog overlays missing modal plumbing** — `apps/sim/src/lib/panels/ResetConfirm.svelte:18-29` and `apps/sim/src/lib/panels/KeybindingsHelp.svelte:33-65`. Both declare `role="dialog" aria-modal="true"` but neither moves focus on open, neither traps Tab, neither handles Escape at the overlay layer, neither restores focus on close. `ResetConfirm` further relies on page-level `Y`/`Esc` listeners, which breaks if the user tabs into a button. These patterns will be cloned as the sim and future surfaces add modals.
5. **Form-submit `enhance` handlers do not move focus on error** — `apps/study/src/routes/(app)/memory/new/+page.svelte:98-104`, `apps/study/src/routes/(app)/plans/new/+page.svelte`, `apps/study/src/routes/(app)/reps/new/+page.svelte`, `apps/study/src/routes/login/+page.svelte:44-50`. When the server returns `fieldErrors`, the page re-renders but focus stays on the submit button (which may now be re-enabled). The error `<span role="alert">` under each field fires, but the first invalid field is never focused. This is a WCAG 3.3.1 miss everywhere forms exist. Fix pattern: on server-returned errors, focus the first invalid input (query by id from `fieldErrors` keys) in `return async ({ update }) => { await update(); focusFirstError(); }`.

## Issues

### CRITICAL: Focus ring does not meet 3:1 non-text contrast

- **File**: `libs/themes/tokens.css:120-122`
- **WCAG**: 2.4.11 Focus Not Obscured (Minimum) / 1.4.11 Non-text Contrast
- **Problem**: `--ab-color-focus-ring: rgba(37, 99, 235, 0.35)` renders as ~#b6c9f8 on white surface and ~#9db8f0 on surface-sunken. Measured against white: contrast ~1.9:1. Every primitive (`Button`, `TextField`, `Select`, `StatTile`, `ConfidenceSlider`, `ConfirmAction`, nav `.identity summary`, login dev buttons, `HelpSearch`) uses this as the sole focus indicator after setting `outline: none`. Keyboard users on WebAIM-default settings cannot see where focus is, especially on `surface-sunken`. Also fails against forced-colors / Windows High Contrast because `box-shadow` is stripped there.
- **Fix**: (a) raise the ring to a solid 3:1+ color: `--ab-color-focus-ring: #1d4ed8` (primary-hover, already used) and keep the softer glow as a secondary `--ab-shadow-focus-glow`. (b) In the primitive styles, use `outline: 2px solid var(--ab-color-focus-ring); outline-offset: 2px;` instead of `outline: none; box-shadow: 0 0 0 3px ...;`. `outline` survives `forced-colors` and is reflected by SystemColor. (c) Add `@media (forced-colors: active) { :focus-visible { outline: 2px solid CanvasText; } }` to tokens.css.

### CRITICAL: Sim `ResetConfirm` and `KeybindingsHelp` dialogs are not actually modal

- **File**: `apps/sim/src/lib/panels/ResetConfirm.svelte:18-29`, `apps/sim/src/lib/panels/KeybindingsHelp.svelte:33-65`
- **WCAG**: 2.4.3 Focus Order, 2.1.2 No Keyboard Trap (inverted — no keyboard *containment*), 3.2.1 On Focus
- **Problem**: Both render `role="dialog" aria-modal="true"` but (1) do not move focus into the dialog on open — a keyboard user opening `?` keybindings help has focus still on the previously-focused control behind the overlay; (2) do not trap Tab inside the dialog, so tabbing reaches cockpit controls that are visually hidden behind the backdrop; (3) `ResetConfirm` expects page-level `Y`/`Esc` listeners rather than dialog-scoped handlers — if the user tabs to the Cancel button, Esc is captured by the page handler (fine) but if they click Cancel then immediately press Y, the page has no focus owner and the handler fires on the newly-unmounted dialog; (4) neither restores focus to the opener on close; the `close` button (`KeybindingsHelp.svelte:38`) uses a literal `x` character, making it unannounced in some AT (relies on `aria-label="Close help"` which is correct, but the visible glyph is ambiguous).
- **Fix**: (a) on `open=true`, `tick()` then `dialogRef.focus()` on the panel or on the Cancel button for `ResetConfirm`. (b) add a `onkeydown` on the overlay that traps Tab inside a computed focusable list (or use the `<dialog>` element + `showModal()` — ships real trap + Esc handling). (c) on close, call back into the page to restore focus to the opener (pass `triggerEl`). (d) replace the literal `x` with `×` (U+00D7) or an SVG + `aria-hidden="true"`.

### CRITICAL: Memory card review — rating buttons distinguish severity only by color

- **File**: `apps/study/src/routes/(app)/memory/review/+page.svelte:574-588`
- **WCAG**: 1.4.1 Use of Color
- **Problem**: The four rating buttons (Again / Hard / Good / Easy) share identical markup and geometry; the only distinction between them is `.rating-label` color (`danger-hover`, `warning-hover`, `success-hover`, `primary-hover`). A user with deuteranopia/protanopia sees four visually identical buttons, and a monochrome display (or forced-colors mode) collapses them to one. Keyboard shortcut hints (1/2/3/4) help, but hints are optional learning; the primary UI must carry the information.
- **Fix**: add a non-color affordance: a leading icon / glyph (`⟲` for Again, `⚠` for Hard, `✓` for Good, `✓✓` for Easy), or the review-interval hint rendered larger (`< 1m`, `< 10m`, `~ days`, `~ week+` already exists as `.rating-hint` — promote it), or a left border-color strip paired with a non-color pattern.

### MAJOR: Form submission does not focus first invalid field

- **File**: `apps/study/src/routes/(app)/memory/new/+page.svelte:98-104`, `apps/study/src/routes/(app)/plans/new/+page.svelte`, `apps/study/src/routes/(app)/reps/new/+page.svelte`, `apps/study/src/routes/login/+page.svelte:44-50`, `apps/study/src/routes/(app)/memory/[id]/+page.svelte`
- **WCAG**: 3.3.1 Error Identification, 3.3.3 Error Suggestion
- **Problem**: `use:enhance` updates the form, the `<span role="alert">` under each invalid field announces, but keyboard focus stays on the submit button. Users then Shift+Tab through every field to find the first error. Errors are visible and announced in principle, but in practice the announcements stack and the user is disoriented. `login/+page.svelte` is particularly bad — the `Banner variant="danger"` error announces the form-level message but focus remains on "Sign in", so a SR user hears the error and then has to Tab backward to reach the email field.
- **Fix**: add a helper that, after `update()`, queries the DOM for `[aria-invalid="true"]:not([disabled])`, picks the first, and focuses it. Wire into every form's `use:enhance` return callback. Consider promoting to `libs/utils` as `focusFirstInvalid(formEl)`.

### MAJOR: `ConfirmAction` tab trap leaks to page when form has hidden inputs

- **File**: `libs/ui/src/components/ConfirmAction.svelte:80-101`
- **WCAG**: 2.1.2 No Keyboard Trap (inverted — should contain, does not reliably)
- **Problem**: The trap only switches focus when `event.target === confirmEl` (Shift+Tab) or `event.target === cancelEl` (Tab). When `formAction` is set, the Confirm button lives inside a `<form>` with hidden `<input>` fields (lines 124-128). Hidden inputs are not focusable, so this is fine in practice, but nothing prevents a future caller from passing visible children via `children` snippet, or a SvelteKit-enhanced form from adding a CSRF token input with `tabindex="0"`. Also: no focus containment on the Cancel->(nothing)->Confirm direction if Cancel is the first focusable; Shift+Tab from Confirm jumps to Cancel (correct), but Tab from Cancel jumps to Confirm only when `target === cancelEl` — if a nested focusable element steals focus between, the trap releases.
- **Fix**: use a single keydown handler that computes `focusableElements = panel.querySelectorAll('button, [href], input:not([type=hidden]), select, textarea, [tabindex]:not([tabindex="-1"])')` and wraps on first/last rather than hard-coding two buttons. Also handle click-out: on `pointerdown` outside `panel`, cancel and restore focus to `triggerEl`.

### MAJOR: `TextField` and `Select` do not expose a required indicator to AT

- **File**: `libs/ui/src/components/TextField.svelte:54-60`, `libs/ui/src/components/Select.svelte:50-56`
- **WCAG**: 3.3.2 Labels or Instructions
- **Problem**: The visual `*` marker next to a required label is `aria-hidden="true"`, and `required` is passed to the input — browsers expose this via `aria-required` implicitly, so it's mostly fine. But `aria-required` is NOT set explicitly, and some older AT combos (NVDA + Firefox pre-2023) did not expose the implicit `required` until focus. Nit-level if you accept the implicit required exposure; major because it's a primitive.
