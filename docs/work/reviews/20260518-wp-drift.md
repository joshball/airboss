# WP Drift Report: 2026-05-18

Spec-vs-code drift check across the work-package corpus (`docs/work-packages/`).
This is a current-state review: documented intent vs shipped code, not a branch diff.

## Method and scope

113 WP directories were inventoried. Frontmatter `status` breaks down as:

| status     | count |
| ---------- | ----- |
| signed-off | 49    |
| in-flight  | 35    |
| draft      | 29    |

`draft` WPs were excluded -- their specs describe planned, not shipped, work, so
path mismatches are expected, not drift. The check focused on `signed-off` and
`in-flight` WPs.

### Why the finding count is low

A path-existence sweep across every spec/tasks/design doc flagged ~130 backtick'd
`libs/...` / `apps/...` paths that no longer resolve. Each was inspected in
context. The overwhelming majority are **not drift**:

- **Deliberate history references.** `bc-citations-coupling/spec.md` references
  `libs/bc/citations/src/citations.ts` throughout its "History" section -- the
  whole WP is the decision to retire that package. The path is *supposed* to be
  broken.
- **`Move X -> Y` task descriptions.** `extract-hangar-bc/tasks.md` has
  `Move libs/db/src/hangar.ts -> libs/bc/hangar/src/schema.ts`. The source path
  is meant to no longer exist.
- **Draft-era hedges in in-flight specs.** `content-citations/spec.md:152`:
  "Read functions in `libs/bc/study` (or a new `libs/bc/citations`)" -- an open
  question recorded before the decision landed.
- **Cross-repo port-source paths.** `hangar-review-queue/spec.md:281` lists
  `libs/constants/src/tasks.ts` explicitly as an `airboss-firc` port source.
- **Truncated illustrative paths.** `apps/ops/.../users/invite/`,
  `apps/hangar/src/routes/cards/...` -- ellipses, not real paths.

Because of this noise, no path references were rewritten in bulk. Only drift
verified by reading both the doc and the code is reported below, and only
genuinely mechanical drift was fixed inline.

## Feature: hangar-platform-dashboard

Work package: `docs/work-packages/hangar-platform-dashboard/`

### Spec Drift

#### MAJOR: WP marked signed-off but the feature does not exist in code

- **Spec says**: a `/platform` command center inside the hangar with panes
  `NowBar.svelte`, `DriftPane.svelte`, `WpStatusBoard.svelte`, etc. under
  `libs/ui/src/platform/`; a BC at `libs/bc/hangar/src/platform.ts`; routes
  `/platform`, `/platform/wp`, `/platform/roadmap`, `/platform/adr`,
  `/platform/coverage`, `/platform/drift`; constants at
  `libs/constants/src/platform.ts`.
- **Code does**: none of it exists. No `/platform*` route under
  `apps/hangar/src/routes/(app)/`, no `platform.ts` in `libs/bc/hangar/src/` or
  `libs/constants/src/`, no `NowBar.svelte` / `DriftPane.svelte` anywhere.
  `tasks.md` shows 0 of 51 tasks checked.
- **Classification**: review needed
- **File**: `docs/work-packages/hangar-platform-dashboard/spec.md` (frontmatter
  `status: signed-off`)
- **Why review**: a `signed-off` WP whose entire implementation is absent is
  either (a) the status was advanced in error, or (b) the feature was descoped
  and the WP should be moved to a deferred/abandoned state. The agent cannot
  flip `status` (lint-blocked; the shipped transition requires human
  `human_review_status: signed-off`). The frontmatter is internally
  inconsistent: `status: signed-off` alongside `human_review_status: pending`.

#### NIT: status/human_review_status inconsistency

- **Spec says**: `status: signed-off`, `human_review_status: pending`.
- **Code does**: n/a -- this is a frontmatter contract issue. Per ADR 025 the
  shipped/signed-off state should follow `human_review_status: signed-off`.
- **Classification**: review needed
- **File**: `docs/work-packages/hangar-platform-dashboard/spec.md:7-8`

## Cross-cutting: signed-off WPs with all-unchecked tasks.md

Several `signed-off` WPs have `tasks.md` files with **zero** checked boxes
despite the feature having shipped: `hangar-platform-dashboard` (0/51 -- but see
above, this one genuinely did not ship), `reference-renderer-runtime` (0/80),
`knowledge-graph` (0/97), `spaced-memory-items` (0/47).

This was *not* fixed by bulk-checking. Checkbox usage is inconsistent across the
corpus -- `calibration-tracker` (17/21 checked) and `decision-reps` (30/33) use
checkboxes and leave a few genuinely-unshipped items unchecked, while others
(`auth-names`, `spaced-memory-items`) ship without ever ticking a box. With no
reliable per-task "did this ship" signal, mass-ticking 275 boxes would be
guessing, and a wrongly-checked box is worse than an unchecked one (it hides
unfinished work). Surfaced here as a tracking-hygiene item for a human pass.

- **Classification**: review needed
- **Severity**: minor

## What was fixed inline

Nothing was rewritten in the specs. Every candidate path mismatch was either
(a) deliberate (history references, `Move X -> Y` tasks, port-source paths,
draft hedges) and therefore not drift, or (b) genuine drift that needs a human
decision (architectural evolution, status transitions the agent is lint-blocked
from making, test-coverage gaps that need confirmation).

The skill's own rule applies: "Never silently update a spec -- every doc change
must be traceable to a drift finding," and forcing a "rename" edit on
exploratory draft text or on architectural drift would corrupt the spec's
recorded intent. The result of this run is the report itself plus a clean
punch list.
