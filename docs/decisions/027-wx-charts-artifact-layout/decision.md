# ADR 027: wx-charts and wx-engine artifact layout

The on-disk layout for chart artifacts under `data/charts/wx/` is reorganized into three families with distinct slug shapes, and every slug + path is constructed through helpers in `libs/constants/`. This ADR records the layout, the centralization rule, and the May 2026 Option A basemap fix that lands in the same overhaul.

## Decision

### Three families under `data/charts/wx/`

Today, 20 wx-charts reference fixtures and 102 wx-engine scenario charts share the same flat root. They are different kinds of artifacts with different lifecycles, and the flat layout hides that. The new layout splits them into three families:

| Family             | Disk path                                                 | Slug shape                                          | Lifecycle                                                                                                                  |
| ------------------ | --------------------------------------------------------- | --------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------- |
| reference-fixtures | `data/charts/wx/reference-fixtures/<lib-slug>/`           | `reference-fixtures/wx-<chart-kind>-<date-zulu>`    | Hand-traced from real WPC/AWC archives. One per renderer. Visual baseline for the 17 chart renderers in `libs/wx-charts/`. |
| wx-scenarios       | `data/charts/wx/wx-scenarios/<scenario-id>/<chart-kind>/` | `wx-scenarios/<scenario-id>/<chart-kind>`           | Generated by `bun run wx-scenario build`. Six scenarios with ~17 charts each. Regenerates on engine change.                |
| mockups            | `data/charts/wx/mockups/YYYY-MM-DD-<short-name>/`         | n/a (not addressable via slug helpers; agent-owned) | Internal design exploration. Not consumed by any production code path. The agent that generates them owns the lifecycle.   |

The slug is the externally-visible identifier consumers use via `<CourseStepChart slug="..." />`. Path-shaped slugs (family prefix segments) make the family obvious at the call site and let the listing code walk each family with its own glob.

Inside any chart directory, filenames carry the slug tail so a stray file on disk is self-identifying. The rule is symmetric across both production families:

```text
data/charts/wx/wx-scenarios/frontal-xc-march/surface-analysis/
  frontal-xc-march-surface-analysis-spec.yaml
  frontal-xc-march-surface-analysis-chart.svg
  frontal-xc-march-surface-analysis-meta.json

data/charts/wx/reference-fixtures/wx-surface-analysis-2024-12-23-12z/
  wx-surface-analysis-2024-12-23-12z-spec.yaml
  wx-surface-analysis-2024-12-23-12z-chart.svg
  wx-surface-analysis-2024-12-23-12z-meta.json
```

Reference-fixtures originally landed in PR 3 with bare artifact names (`spec.yaml`, `chart.svg`, `meta.json`) because the directory already encoded the slug. The follow-on disambiguation pass renamed all 60 files (20 dirs x 3 artifacts) to apply the same rule the wx-scenarios family already used: filename-alone identifies the source slug, so accidental cross-directory copy-paste cannot silently corrupt the wrong fixture.

### Centralization rule

All chart slugs and chart-artifact paths flow through helpers in `libs/constants/`:

- `libs/constants/src/wx-engine-paths.ts` for the wx-scenarios family
- `libs/constants/src/wx-charts-paths.ts` for the reference-fixtures family and the shared root-dir helper

No inline templates anywhere else in the codebase. Adding a new chart kind = update the `CHART_TYPES` / scenario enum and the helper picks up the new shape. The follow-on directory rename is a one-line helper change instead of grep-and-replace across 20+ sites.

`scripts/charts/lib.ts:chartsDir()` is retained but reimplemented to call `chartsRootDir()` from `@ab/constants`. Callsites in scripts and library code import the helpers from `@ab/constants` directly; the legacy `chartsDir()` shim gives the build CLI a stable surface to import from while the rest of the codebase converges on the shared helpers.

### Mockups carveout

`data/charts/wx/mockups/` is internal-only. No helpers, no production consumers. Mockups are generated by design-exploration agents (basemap A/B/C explorations, projection comparisons, layout sketches), reviewed, and either:

- Promoted to permanent reference material (moved to `reference-fixtures/`, given a proper `wx-<kind>-<date-zulu>` slug), or
- Cleaned up via session-end housekeeping.

Mockup paths are agent-owned. They are not stable identifiers and `<CourseStepChart>` never points at them.

### Option A basemap (cross-reference)

The wx-charts CONUS basemap is extended with Canada + Mexico outlines (Option A from the May 2026 basemap exploration). This fixes the "hunchback" appearance of the Lambert Conformal 33/45 projection on charts that crop to CONUS: without context to the north and south, the projection's curvature on the top/bottom edges reads as a deformation of the United States outline rather than the natural distortion it actually is.

The basemap fix lands in PR 2 of this overhaul (separate from the helper plumbing in PR 1). Implementation is out of scope for this ADR; the ADR records the decision so it is the single source of truth for the May 2026 chart-layout overhaul.

## Migration

Three PRs:

1. **Helpers + inline-template removal (this ADR ships with PR 1).** No file moves. `libs/constants/src/wx-engine-paths.ts` and `libs/constants/src/wx-charts-paths.ts` introduced; the 13 wx-engine chart files and the date-stamped reference-fixture tests stop hand-rolling slugs and call the helpers; `scripts/charts/lib.ts` and the scenario build script route through `@ab/constants`. Helpers return the **current flat layout** so the engine bundle writer and the wx-charts tests are byte-identical to the previous behavior. Tests cover every helper.

2. **wx-charts basemap fix (Option A).** Adds Canada and Mexico outlines to the basemap pipeline. Regenerates the 20 reference-fixture SVGs. No code-shape change to the chart families.

3. **Layout migration.** Moves the 20 reference-fixtures into `data/charts/wx/reference-fixtures/`. Moves the 102 wx-scenarios into `data/charts/wx/wx-scenarios/<scenario-id>/<chart-kind>/`. Updates the helpers in `libs/constants/wx-*-paths.ts` to compute the new nested shape. Updates `scripts/charts/lib.ts:listChartSlugs` to walk both families. Regenerates the 102 scenario charts via `bun run wx-scenario build --all` (idempotency check confirms byte-equality where applicable). Reruns the wx-charts reference-fixture tests to confirm the helper-derived slugs match the new disk paths.

Because every slug and path goes through `@ab/constants` after PR 1, PR 3 is a one-line helper change plus the `mv` operations. No grep-and-replace across the codebase.

## Out of scope

- Render-time changes to chart appearance other than the basemap fix (separate WPs handle individual chart improvements).
- Migration of the wx-scenarios bundle layout under `data/wx-scenarios/<slug>/` (truth + products + commentary). That tree is unaffected by this ADR; only the chart-artifact mirror under `data/charts/wx/` is reorganized.
- Schema or DB changes. There are no DB rows for chart artifacts; the slug is the addressable identifier and the file path is derived from it.
