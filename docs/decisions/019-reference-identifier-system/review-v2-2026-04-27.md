# Review of ADR 019 v2 -- Reference Identifier System

## Summary

**Verdict: approve with revisions.** The v2 is materially better than v1 and addresses most of the round-1 concerns substantively, not cosmetically. The four-move spine (identifier-as-contract, registry-as-resolver, body-as-source-of-truth, five-tier storage) is now stronger because the per-corpus opacity (§1.1, §1.2) and the registry's three-concern split (§2) eliminate the worst grammatical brittleness of v1.

That said, three corners are still being painted. (1) The colon separator (§1.1) collides with URI-scheme parsers in non-airboss tools without the ADR confronting it. (2) The supersession-acknowledgment syntax (§6.3) hides structured state inside a Markdown link `title` attribute, which is a fragile carrier that rotates through editors and renderers and won't survive contact with copy-paste. (3) The auto-stamp pinning rule (§1.3) writes to author files mid-validation, which is correct in spirit but creates new failure modes the ADR doesn't address (stale registry, agent-generated content, copy-paste between lessons). These three should be resolved before approval, not after Phase 1 ships.

There are also a handful of internal-consistency drifts (the "retired" lifecycle is referenced in §9 but not declared in §2.4; "indexed tier" cross-references contradict between §2 and §4; the §13 changelog table has at least three errors) that should be cleaned up before the document moves out of draft.

## Critical concerns (new in round 2)

### C1. The colon separator collides with URI-scheme parsers and the ADR doesn't confront it

§1.1 changes the corpus-locator separator from `.` to `:` -- a clean fix for the dot-collision problem. But `<corpus>:<locator>` is the exact shape of a URI scheme. `regs:cfr-14.91.103@2026` will be interpreted by *every* URI parser in the wild as scheme `regs`, scheme-specific-part `cfr-14.91.103@2026`. This causes:

- **CommonMark autolinks.** GitHub, GitLab, eCFR's own viewer, anything Markdown-rendered outside our renderer will try to make `regs:cfr-14.91.103@2026` clickable as a URL, fail (no registered handler), and either render it as a broken link or strip it.
- **Browser address bar.** Pasting `regs:cfr-14.91.103@2026` into Chrome will trigger "do you want to register a handler for `regs:`?" or a search engine fallback, depending on the browser.
- **IDE link detection.** VS Code's link provider sees `corpus:locator` and tries to dispatch to a registered URI handler.
- **Slack/Discord/iMessage unfurl.** The unfurl agent treats `regs:...` as a URL with scheme `regs:`, fails to fetch, shows nothing useful (the §3.1 table claims "Title from `@cite`" but that requires a renderer that isn't there).
- **`mailto:`-style ergonomics.** Some users will paste these into a browser expecting them to resolve. With no scheme handler registered, they get nothing.

§3.1's row "RAG citation footnote" preserves the identifier as machine-readable; §3.1's row "Slack / iMessage unfurl" claims `@cite` extraction. These both rely on the airboss renderer being in the path. When it isn't (which is most external contexts), the URI looks malformed.

The §9 entry "Lesson rendered in non-airboss Markdown viewer (GitHub)" claims the resolver service makes external viewers see useful URLs. That's not what happens. External viewers see `regs:...`, parse it as a URI, find no registered scheme, and render it as text or a broken link. The resolver-service URL only works if the lesson body contains `https://refs.airboss.dev/...` -- but the body contains `regs:...`. The §9 entry conflates "we can render to a useful URL" with "the body URL is useful."

**Why it matters:** the v1 review's concern about "plain Markdown" was claimed-fixed in v2 by removing the multi-reference microsyntax. But the `<scheme>:<rest>` shape itself is the new violation. We're shipping content with non-resolvable URI-shaped strings. Five years from now, anyone reading the lesson markdown outside our renderer (a contributor, an LLM ingesting via RAG, an `archive.org` snapshot, a backup-restore PDF, the user's own repo browser when grepping) sees identifier-shaped non-URLs.

**What to do:** pick one and commit:

- **Register the scheme:** `airboss-ref:regs/cfr-14/91/103?at=2026`. URL-shaped, parseable by every URI library, no collision risk. Loses the readability of `regs:cfr-14.91.103@2026` but gains correctness.
- **Use a custom delimiter that's not URI-scheme-shaped:** `regs/cfr-14.91.103@2026` (slash instead of colon). Still readable; doesn't trigger URI-scheme parsing.
- **Rewrite at write-time, not at render-time:** the validator transforms `regs:...` to a real URL when the lesson is saved. Body always contains real URLs; identifier shape is virtual. (Author convenience plus external compatibility, but harder to grep.)
- **Accept and document:** add a §1.1 subsection that explicitly says "these are not URIs; they will look broken in non-airboss viewers; this is a deliberate trade-off; here is how to disambiguate from URI parsing." At minimum, the ADR should *acknowledge* the collision instead of pretending it isn't there.

The current ADR pretends the colon is unambiguous. It is unambiguous to *our parser*; it is exactly ambiguous to every other parser.

### C2. The auto-stamp validator rewrites author files mid-build, which fails several real workflows

§1.3 says: "Authors may write unpinned (`regs:cfr-14.91.103`); the validator auto-stamps `@<current>` on first save and rewrites the body. After validation, every identifier in the body is pinned. The author sees the auto-stamp in the PR diff and can override."

This is the right intent (no unpinned references in committed lessons). But the mechanism has problems:

1. **Agent-generated content.** Claude or a script writes a lesson, commits, pushes, opens a PR. The validator runs in CI. CI doesn't have write access to author the auto-stamp commit (or if it does, that's a different security problem). Either CI fails (because unpinned references didn't get stamped) or CI commits back to the branch (race condition with the human's next commit, surprise diff in their working copy).
2. **Copy-paste round-trip.** Author copies `[@cite](regs:cfr-14.91.103@2026)` from lesson A into a chat with the user; user pastes it into lesson B. Now lesson B has `@2026`, which auto-stamp would never overwrite (it only stamps unpinned). But lesson B was authored in 2030 -- the pin is stale-but-valid. The validator doesn't warn (per §1.5: "If pinned edition is older than current: success, no warning during normal validation"). Lesson B silently ships stale.
3. **Stale registry at validation time.** Author pulls main; ingestion has not yet promoted `pending` -> `accepted` for the latest edition. Validator auto-stamps to whatever is currently `accepted` (last year's edition). Author's PR ships with last year's pin. Six months later, ingestion promotes; the lesson is now obviously stale but the diff job (§5) only fires on hash-differs, not on "this lesson was pinned to N-1 when N was available."
4. **Mid-PR registry change.** Author writes lesson against `accepted` 2026 entries. While their PR sits in review, ingestion lands 2027 and promotes it to `accepted`. Their next push triggers re-validation; auto-stamp now wants `@2027` instead of `@2026`. The PR diff suddenly contains every reference re-stamped. Confusing.
5. **Multi-machine workflow.** User edits a lesson on laptop A (auto-stamps `@2026`). Pulls on laptop B which has a slightly different registry state (different ingestion run). Validator on B re-stamps. Working tree diverges from origin; user sees noise commits.
6. **The auto-stamp-then-author-overrides flow.** §1.3 says "the author sees the auto-stamp in the PR diff and can override." Override how? By manually editing the body? Then the validator re-stamps on next save? Is there an opt-out marker that says "I deliberately want unpinned"? §1.3 doesn't say.

**Why it matters:** the auto-stamp turns the validator into a write-side actor. Validators that mutate source are a known footgun (think `gofmt --in-place` semantics in CI, or the historical pain of `prettier --write` running in pre-commit hooks). The pattern is workable but only with explicit rules for who runs it where.

**What to do:**

- Specify *where* the auto-stamp runs: `bun run check` (developer machine) only, never in CI.
- CI runs a *check* mode that fails if unpinned references exist, instructs the author to run `bun run check` locally, but does not write.
- Specify the override marker. E.g. `[<text>](regs:cfr-14.91.103@unpinned)` is a deliberate explicit unpin, validated as WARNING, not auto-stamped.
- Specify the staleness rule: at auto-stamp time, the validator records the registry's `accepted` state hash; on subsequent validation runs, if the stamp is older than the current `accepted` state by more than N editions, emit WARNING (linking to the diff job). Otherwise the lesson silently ages out per (3) and (5) above.
- For copy-paste from chats / external sources, accept that the user is responsible. But document it in §1.3.

### C3. Supersession acknowledgment hides structured state in a Markdown link `title` attribute

§6.3 introduces:

```markdown
[Mangiamele letter (2009)](interp:chief-counsel.mangiamele-2009 "+ack:smith-2027:original-intact")
```

The link `title` slot (the third quoted argument in Markdown link syntax, conventionally rendered as a hover tooltip in HTML) is now load-bearing structured data. The validator parses it for `+ack:<id>:<reason>`. Problems:

1. **Editor mangling.** Markdown editors that "tidy" links (Obsidian, Bear, Notion when pasted, Marked, even some VS Code extensions) routinely strip or rewrite link titles. A round-trip through one of these silently destroys the acknowledgment.
2. **CommonMark renderers vary.** GitHub's renderer hides link titles (no hover tooltip on the web). They survive in source but are invisible to readers, including PR reviewers. A reviewer reading a PR that adds an `+ack` cannot see it without opening the raw source.
3. **Quote and backslash escapes.** Reason slugs are "free-form short strings." If a reason contains `"` or `\`, Markdown link-title parsing fails. The ADR doesn't specify the escape rule. (Would `+ack:smith-2027:rule-still-applies-because-\"common-purpose\"-test` work?)
4. **Two annotations on one link.** What's the syntax for `+ack:X:R` AND `+historical`? `+ack:X:R+historical`? `+ack:X:R;+historical`? Multi-line title?
5. **Chain of supersessions.** §6.3 says the `+ack` "carries the superseding entry it acknowledges (so a *future* supersession of the superseder re-triggers warnings)." But the ack format is `+ack:<superseding-id-shortform>:<reason>`. What if the superseder is itself superseded? Now the lesson's `+ack` points at a now-also-stale entry. Author has to update the `+ack` reference. The §6.3 mechanism assumes one level of supersession; reality has chains.
6. **Diff hostility.** A renumbering where the superseder's id changes (e.g. the Walker letter gets re-cataloged as `interp:chief-counsel.walker-2017.v2`) requires every `+ack` referencing it to be updated. There's no reverse-index in §2.3 of "ack annotations referencing this id" -- so finding them requires grepping lesson source.
7. **Acknowledgment-of-historical conflict.** What does `[text](id "+ack:X:R+historical")` mean? Both? Just historical? §6.3 lists A/B/C as alternatives but doesn't say they compose.
8. **Non-Markdown contexts.** §9 says "scenario fixture file (TypeScript, not Markdown)" uses identifier constants -- which has no link-title slot. So acknowledgments only work in Markdown. Lessons in non-Markdown surfaces (TypeScript fixtures, future YAML lesson formats, audio transcripts) have no acknowledgment mechanism.

**Why it matters:** the acknowledgment is the user's explicit ask -- they want to retain references to original sources with annotation. Hiding the annotation in the link title makes the annotation invisible to readers, fragile to editors, and untyped. It will not survive contact with five years of authoring.

**What to do:**

- Move acknowledgments out of the link title and into a structured place. Options:
  - **Frontmatter block.** Lesson frontmatter gets an `acknowledgments:` array of `{ id, ack: { superseder, reason }, historical?: bool }` records. Frontmatter is structured, diff-friendly, queryable, lossless through editors.
  - **Companion comment block.** Markdown extension: `<!-- @ack id=interp:chief-counsel.mangiamele-2009 superseder=smith-2027 reason=original-intact -->` immediately preceding the link. Visible in source, diff-friendly, survives editors.
  - **Sibling registry entry.** The lesson references `interp:chief-counsel.mangiamele-2009` as normal; an entry in the lesson's frontmatter binds it to an acknowledgment. The same lesson can have multiple references to the same entry; each binding is keyed to a per-link anchor.
- If keeping the title-slot syntax, document the escape rules, the multi-annotation grammar, and the chain-resolution behavior. None of these are in §6.3 today.

### C4. Phase 1 (validator) before Phase 4 (CFR ingestion) means the validator's resolution check is a no-op for the longest stretch of the work

§8 ordering: Phase 1 (validator), Phase 2 (registry-core lib, *empty*), Phase 3 (renderer), Phase 4 (CFR ingestion), Phase 5+ (other corpora). The §8 footnote says "Authoring lessons against any new corpus requires that corpus's ingestion to have landed."

So between Phase 1 landing and Phase 4 landing, the validator is operational but has nothing to validate against. Per §1.5, a non-resolving identifier is an ERROR. With an empty registry, every identifier fails. Three options:

- The validator has a "registry empty" mode that downgrades resolution failures to WARNINGs. Not specified.
- Authoring is paused between Phase 1 and Phase 4. Three phases of "no new lessons may be written." Not realistic; the user has authoring momentum on the regulations course.
- Authors use `unknown:` for everything. But §1.6 says `unknown:` "cannot be published" and is "explicitly the unhappy path." If the entire CFR corpus is an unknown for three phases, the unhappy path is the path.

Per (C5 in round 1), the v2 §13 claim is "bulk ingestion first; `unknown:` for transitions." The v2 §8 ordering shows bulk ingestion (Phase 4) is *not* first; Phase 1 (validator) is first. The §13 claim about phasing is not what §8 actually does.

**Why it matters:** lesson authoring during the validator-without-registry gap is exactly the corner the user said they want to avoid. They're either blocked, working in `unknown:` (which has its own publish gate per §1.6), or relying on the renderer existing before the validator is meaningfully useful.

**What to do:**

- Move CFR ingestion (Phase 4) immediately after the registry-core lib (Phase 2), before the renderer (Phase 3). Order: ADR -> validator-shape (no-op resolution for empty registry) -> registry-core -> CFR ingestion -> renderer -> versioning -> ...
- Or: Phase 1 + Phase 2 + Phase 4 land together as one milestone ("validator + registry + first corpus, lessons may now author against `regs:`"), and Phase 3 (renderer) follows.
- Specify the validator's "registry-not-yet-populated-for-corpus-X" behavior explicitly. Either ERROR (and lessons can't reference X), or WARNING (and lessons may), or `unknown:` is the only valid form pre-ingestion. Pick one.

### C5. Acceptance criteria are checkbox theater

§11 has 9 checkboxes. They check: was a section reviewed, does a section list every X, was something updated, no `TBD` anywhere. None of these are unambiguous "this passes" conditions:

- "§1 reviewed against all corpora named in the reviewer charter" -- reviewed by whom, judged how?
- "§1.5.1 lists every validation case and assigns a tier" -- the list is bounded by what someone thought to enumerate. Easy to miss cases.
- "§3.1 covers every render mode named in S8" -- the v1 review's S8 doesn't have a closed list of render modes; it names "at least: print, audio TTS, screen reader, plain-text export, RSS, share-card." v2 adds five more. Someone still has to judge "is this exhaustive enough"?
- "No `TBD` anywhere" -- easy to satisfy by deleting `TBD` and writing a sentence that doesn't commit to a rule. Same effect, undetected.

There are no concrete "this artifact must exist" or "this command must produce this output" criteria.

**Why it matters:** the user explicitly cited the round-1 nit about acceptance criteria. The v2 has criteria, but they don't gate anything except "did somebody nod yes." The criteria for ADR 018 are concrete: "§file X exists, line Y of file Z reads ABC." Match that pattern.

**What to do:** rewrite §11 with concrete, machine-checkable items. Examples:

- `libs/sources/src/index.ts` exports `resolveIdentifier`, `hasEntry`, `getChildren`, `walkSupersessionChain`, `findEntriesByCanonicalShort`, `findEntriesCitingFromLessons`, `findLessonsCitingMultiple`, `getCurrentEdition`, `getEditions`, `isPinStale`.
- `STORAGE.md` "The five tiers" table contains five rows: source / derivative / indexed / computed / generated.
- `bun run check` exits non-zero on a fixture lesson with an unresolvable identifier; exits zero on a fixture lesson with a resolved-pinned identifier.
- The `+ack` syntax (or its replacement) round-trips through `<chosen-editor>` without loss in a fixture test.
- A scripted test imports `@ab/sources` from a non-Svelte CLI tool and resolves an identifier.

## Significant concerns (new in round 2)

### S1. The `unknown:` escape hatch can leak past CI

§1.6 says `unknown:` references "cannot be published. CI gates published lessons against any `unknown:` references." But:

- "Published" is not defined. Is published `main` branch? Production deploy? PR-merged? A lesson can sit in PR with `unknown:` indefinitely if "publish" only means "deployed." In a feature-branch workflow with several long-lived authoring branches, `unknown:` accumulates.
- The §1.6 properties say "they cannot be published. CI gates published lessons against any `unknown:` references." This implies a separate gate from `bun run check`. Not specified where it lives, what tool runs it, or what triggers it.
- §8 doesn't list a "publish gate" work package.
- If 50 lessons end up with `unknown:` references when an authoring deadline arrives, the gate forces author panic-replacement of all 50 against a registry that may not yet have entries. Same corner the v1 was trying to avoid, just deferred.

### S2. The five-tier model bleeds at the computed boundary

§4 puts "embeddings, full-text indexes, knowledge graph edges, cross-corpus joins" in computed. But:

- Knowledge graph edges that link lesson-to-source-entry: rebuilt when *lesson* changes, not when source changes. Different invalidation than embeddings on the source corpus. Same tier in §4.
- Per-session caches (a lesson-page server load function memoizes a render): per-user, per-session. Where does it live? Computed (because it's derived) or generated (because it's per-user)?
- Per-user search indexes (an Algolia-style "search this user's bookmarks" feature): owned by platform, rebuilt on user activity. Generated by §4. But the index *structure* is owned by platform, like computed. Boundary blur.
- Cross-corpus joins: depending on the join, may be invalidated by *either* of two ingestion runs. The current "Indexed change OR algorithm/schema migration" trigger doesn't capture multi-source invalidation.

The v1 review (C4) flagged this exact split. The v2 fix puts a fifth tier in but doesn't tighten the invalidation rules. Worth a sub-bullet under §4 stating: "Computed-tier artifacts may have multiple invalidation sources; an artifact that depends on K corpora is rebuilt when any of K is invalidated; the computed-tier registry tracks dependencies, not just sources."

### S3. The §2.5 render-time loading strategy is specified for one renderer

§2.5 specifies the lesson-page server load batches identifiers. What about:

- The RAG citation-snippet builder (which renders fragments of lessons in a non-page context).
- The share-card preview generator (1200x630 image; runs at deploy or at request, not in lesson-page context).
- An RSS feed that includes lesson summaries (rendered offline, not in a request context).
- A CLI export tool that produces PDFs (no SvelteKit server load).
- A CLI export tool that produces a complete bibliography for a study guide.
- A search-result snippet (which inlines a partial lesson).

The §3.1 render-mode table covers what each mode *outputs*; §2.5 covers how *one* renderer fetches data. Other renderers each need their own batch-resolve implementation, or a shared "render orchestrator" that all surfaces use. §2 doesn't expose the latter.

If every new render surface re-implements the batch-resolve, we will eventually have N versions with subtle differences. Specify that batch-resolve is exposed by `@ab/sources` itself (`@ab/sources/render`?) and that all surfaces consume it.

### S4. Several token entries in §3.1 are inconsistent or undefined

The render-mode table:

- "TTS: tokens substitute; identifier is omitted from spoken output" -- but `@text` substitutes the *full text* of the section. A TTS rendering of "(full quoted text of 91.103, paragraphs (a) and (b))" reads the entire regulation aloud. Was that intended? The pure render-mode rule says yes; usability says no.
- "Tooltip / hover preview" uses `@cite`, truncated at 200 chars. But the *author* wrote `@text` or `@quote`. Does the renderer override the author's token choice in tooltip mode? Or render the author's choice and just truncate? Different behaviors; not specified.
- "Transclusion: tokens substitute at host-render time, against the host's registry context" -- what if the host has a pinned-to-different-edition context than the included lesson? Does the included lesson's reference to `regs:cfr-14.91.103@2026` re-resolve against host's edition (silently changing what the included lesson says) or against its own pin (consistent but cross-context)? The current "host's registry context" language implies the former; the round-1 mandatory-pinning rule implies the latter.

### S5. The lifecycle table in §2.4 is missing states implied elsewhere

§2.4 declares two states: `pending`, `accepted`. §9 declares a third: `retired`. §2.1's `SourceEntry.lifecycle` types only the first two. Drift.

What about:

- `draft` -- a corpus WP author writing entries before submitting to ingestion. Promoted to `pending` when bulk-loaded.
- `deprecated` -- still resolves but discouraged. Same as `retired`?
- `wrong` -- entry was a mistake. Lessons reference it. Removing it would orphan; keeping it propagates the mistake. What's the recovery state?
- The transition rules: who promotes pending -> accepted? Does `accepted` -> `retired` happen via supersession only, or directly?

Specify the lifecycle as a closed enum with documented transitions (state machine).

### S6. The `unknown:` corpus has structural problems

§1.6 says `unknown:<descriptive-string>` is an escape hatch. Issues:

- Two authors write `unknown:expanded-checklist-rule` and `unknown:expanded-checklist` for the same thing. Now there are two unknowns; promoting them to a real corpus requires manual disambiguation.
- An `unknown:` slug doesn't have to match the future registry slug. So when the registry catches up and the entry is added, every lesson that referenced it via `unknown:` has to be hand-edited to the real id. Mechanical migration is hard because the mapping isn't recorded.
- §1.6 says `unknown:` references parse and resolve; but the only registered "resolver" for them is "the renderer prints the descriptive string + [unresolved] badge." That's not a `CorpusResolver` in the §2.2 sense. So is `unknown:` a corpus or a magic prefix? §1.1's enumeration calls it a corpus. If it's a corpus, it has a resolver. If it's magic, it has special-case validator and renderer logic. Pick one and document it.

### S7. The phase-9 lesson-migration WP has a chicken-and-egg

§8 Phase 9 is "one-pass mechanical migration of pre-ADR-019 lessons to identifier syntax." Pre-ADR-019 lessons (Week 1 capstones, the night-IFR oral) currently reference regulations as plain text and live URLs:

- `[§91.103](https://www.ecfr.gov/...)` -- a link to eCFR.
- `**§91.103**` -- bold prose with no link.
- `91.103` -- bare prose.

Mechanical migration to identifier syntax:

- Bold-prose case has no URL, no link text. Auto-rewrite to what? `[**§91.103**](regs:cfr-14.91.103@<???>)`? Which edition?
- Bare-prose case is actually arguable -- some occurrences are intentional ("the §91.103 family of rules"); mechanical rewrite turns every occurrence into a link, which may be wrong.
- Existing-eCFR-URL case: parse the eCFR URL to extract section, generate identifier. But the URL form has historically had several variations; the parser must be robust.
- Edition pinning: pre-ADR lessons have no pin. Mechanical migration must pin to *what* -- their `last_verified` date? "Current at migration time"? The author's intent at original write time?

Phase 9 is described in one row of §8 with no spec. This deserves its own work package or at least a §-section with the rules. Otherwise the migration produces silently-wrong content.

### S8. Cross-references that span editions

A lesson cites `regs:cfr-14.91.103@2026` and `interp:chief-counsel.walker-2017` (no edition pin -- letters don't have editions in the same sense). The lesson's claim is "Walker interprets 91.103 to require active investigation."

In 2030 the FAA amends 91.103 in a way that changes the standard (active investigation is now codified). The Walker letter is now historically relevant only -- the lesson's claim about Walker still holds for 2017-2030, but the 2030 reader interpreting "the standard" gets the wrong result.

The diff job (§5) catches the 91.103 amendment. Walker isn't flagged because it didn't change. But the *cross-reference* is now stale. The lesson is making a claim about a relationship between two things, and one of them changed.

The ADR has no model for cross-reference staleness. Per-identifier staleness is handled (§5); per-pair-of-identifiers staleness is not.

### S9. The §13 changelog table has at least three errors

- Row "§4 storage tiers" claims v1 had four (source/derivative/indexed/generated). v1 §4 (per `decision-v1.md`) actually presents *four* tiers but explicitly says "ADR 018 currently has three tiers (source / derivative / generated)" and that this ADR introduces a fourth (indexed). So v1 added one tier; v2 adds *two* (indexed AND computed). The §13 row reads as if v1 had it right and v2 added one; it added two relative to ADR 018 baseline.
- Row "§6 aliases: 'info-level notice' -> tiered" -- v1 (per §6 of decision-v1) does describe alias at info-level for renumbering and warning for cross-section. The v1 already has *two* tiers, not one. v2 adds a third (`silent`). So the change is "two tiers -> three," not "info-level -> tiered."
- Row "§9 deferrals: implicit -> explicit per-item table" -- v1 had a §9 "Out of scope" with four items. The v2 explicit table is genuinely new. But "implicit" mischaracterizes v1's "Out of scope" section. More accurate: "v1 had a four-item out-of-scope; v2 has both an out-of-scope (§10) and a 27-row resolved-deferrals table (§9)."

### S10. The §9 deferrals table uses "DEFERRED with trigger" inconsistently

Some rows have crisp triggers ("when first non-English / non-FAA corpus is ingested"). Others are vague:

- "Pub/sub regulation-diff alerts ... deferred to a notifications WP with trigger ('when learner-account features ship')" -- when do learner-account features ship? Not in §8. So the trigger may never fire by accident.
- "Audio transcript spoken-form citations ... Out of scope for v1 of registry. When audio surface ships, the indexed tier adds spoken-form aliases." -- when does audio surface ship? Open-ended.
- "Academic citation formatters ... deferred; trivial extension when needed" -- trigger is "when needed," which is the exact corner the user warned against.

Per CLAUDE.md "no undecided 'considerations for future work,'" each deferral must have a concrete trigger. "When needed" is not concrete.

## Minor concerns / nits (new in round 2)

### N1. `regs:cfr-14.91` is shown in v1 §1.2 but removed in v2

v1 had `regs.cfr-14.91` (whole part) as a valid identifier. v2 removed it. Was this intentional? Whole-part references are rare but real (e.g. "see Part 91 generally"). The §13 changelog doesn't mention the removal.

### N2. The reverse index `findEntriesCitingFromLessons` has an unclear name

"Find entries cited [by lessons containing] from lessons" parses as a typo. Probably meant `findLessonsCitingEntry(id)`. Same for `findLessonsCitingMultiple`. The current names invert subject/object.

### N3. `Edition.last_amended_date` is "non-null; defaults to published_date"

§6.1's `Edition` interface declares `last_amended_date: Date` (not null, not optional). The "defaults to published_date" language is unusual. Either the field is nullable with a default-on-read rule, or it's required at write time and the default is applied by the writer. Distinction matters for ingestion contracts.

### N4. The Edition interface in §6.1 still has `aliases?: AliasEntry[]` but §9 says aliases support `string[]` for branching

§6.1's `AliasEntry.to: string` is single-valued. §9's row "One-to-many alias" says `to` is `string | string[]`. The actual interface should reflect that. Currently they disagree.

### N5. `ENGINE_SCORING` rule and `ROUTES` rule cited in CLAUDE.md should bind here too

The ADR will produce `@ab/sources/tokens.ts` (a token registry) and `@ab/sources/corpora.ts` (a corpus registry) per §3.1 and §2.2. These are constants modules. Per CLAUDE.md "All literal values in libs/constants/," should they live in `@ab/constants` instead? Or is `@ab/sources` justified as a sibling because it's not just constants but also runtime resolvers? Address explicitly so future agents don't relocate the files.

### N6. `last_amended_date` vs `@last-amended` token: do they match exactly?

§3.1 lists `@last-amended`. §6.1 lists `Edition.last_amended_date`. The token says "date the referenced section was last amended." That's the per-section last-amended, not the per-edition. Editions amend many sections; a single edition has one publication date but every section in it has its own last-amended history. The token resolves against per-section history, but the schema field is on `Edition`. Mismatch.

### N7. `aliases?: string[]` on `SourceEntry` is one of two `aliases` fields

`SourceEntry.aliases?: string[]` (e.g. ["Skyhawk"] for c172s) and `Edition.aliases?: AliasEntry[]` (renumbering). Two different "aliases" with different shapes. Confusing. Rename one. Suggestion: `SourceEntry.alternative_names` for marketing/colloquial, keep `Edition.aliases` for renumbering.

### N8. The `airboss-ref:` resolver-service URL example uses `regs:cfr-14.91.103@2026` as the path

§9's row "Lesson rendered in non-airboss Markdown viewer (GitHub)" gives `https://refs.airboss.dev/regs:cfr-14.91.103@2026` as the resolver URL. The `:` in the path is technically valid but routes through every URL parser as a quirky case. Most URL libraries will parse the host as `refs.airboss.dev`, the path as `/regs:cfr-14.91.103@2026`, and that *probably* works -- but `@` in a path component triggers userinfo parsing in some libraries. Test before committing to this URL shape.

### N9. The §12 worked example uses `+ack:smith-2027:scope-narrowed-but-91-103-claim-intact`

The reason slug is hyphen-separated, fine. But it's 47 characters. That's longer than most reason-slugs would naturally be. Document a recommended length or character class. (Compare to N5 in round 1: token vocabulary was the issue; here it's slug-vocabulary in the ack syntax.)

### N10. §1.5 row "Pinned edition is older than current" says "(no signal here -- see §5 diff job)"

The validator emits no signal when a pin is stale. The diff job catches it. But the diff job runs on the *registry*, not on the lesson body, per §5. Stale pins are caught when the *registry* gets a new edition; not when the lesson is freshly authored against a stale pin. So the cycle is: author writes against an old pin (validator silent, no warning), commits, deploys; year later, ingestion lands new edition; diff job rebuilds; reports stale pin. Author's lesson has shipped stale for the intervening year.

Either tighten the validator (warn on pin older than current) or accept the year-of-staleness window.

### N11. §3.1 "Tooltip / hover preview" claims `@cite` truncated at 200 chars

What if the author wrote `@quote`? The renderer overrides? Or just truncates whatever the author wrote? Specify.

### N12. §3.1 "Plain-text export" appends URL as `(<URL>)` after substituted text

Which URL? The identifier (`regs:cfr-14.91.103@2026`)? The resolver URL (`https://refs.airboss.dev/...`)? The live URL (eCFR)? Three different choices, three different paste-target behaviors.

### N13. The §1.4 multi-reference mechanism uses `@list` as a token AND as a renderer-adjacency-merge result

`[@list](regs:cfr-14.91.167@2026)` substitutes a list (per §3.1), but only if there are *adjacent* identifiers from the same corpus and pin (per §1.4). What if a single link uses `@list`? Does it just produce the canonical_short for that one identifier (a list of one)?

### N14. ACS pinning is unspecified

§1.2 shows `acs:cfi-asel.area-i.task-a` with no edition pin example. ACS publications have explicit revision dates (FAA-S-ACS-25 etc.) and update on multi-year cadences. The mandatory-pinning rule (§1.3) requires every identifier to be pinned. What's the ACS pin format? Slug-encoded (`acs:cfi-asel.faa-s-acs-25.area-i.task-a`)? `@`-suffix (`acs:cfi-asel.area-i.task-a@2024`)? Both? §1.2 doesn't say; the validator can't auto-stamp without a rule.

### N15. The `phak.8083-25C` identifier already encodes the edition

§1.2 says "Edition is part of the doc slug" for handbooks. So `handbooks:phak.8083-25C.12.3` is implicitly pinned. But §1.3's mandatory-pinning rule says every identifier is pinned. Does the validator auto-stamp `@<published>` on top, producing `handbooks:phak.8083-25C.12.3@2024-09-15`? Or does the slug-encoded edition satisfy the pinning rule without an `@` suffix?

§1.3's "Slug-encoded editions are *also* pins" partially addresses this, but the auto-stamp behavior in this case is ambiguous: does the validator do nothing (slug already pins), or stamp the publication date redundantly?

## Cases the ADR still doesn't address

These are scenarios that the v1 review listed (or should have) and that v2 either doesn't resolve or resolves shallowly. Each is one-line to flag scope.

1. **A lesson cites a paragraph in an edition that didn't yet exist when the lesson was written, but pins to it.** Future-pin on a not-yet-published edition. Validator says "pinned edition doesn't exist -> ERROR." Author retries. Forever.
2. **Two ingestions run concurrently and conflict.** Annual + mid-year correction land overlapping. §9 says "Latest by date wins" but doesn't define "date" -- publication date? Ingestion date? Last-amended date?
3. **An edition is partially ingested and a section's pending entry promotes while a sibling stays pending.** Validator state varies per-section within the same lesson. Half a lesson resolves; half doesn't. CI is half-broken per-PR.
4. **The auto-stamp lands a pin to an `accepted` edition that subsequently gets rolled back** (engineer review caught a bug post-promotion). Does the lesson's stamp roll back too? §2.4 doesn't define the un-promote path.
5. **A regulation transfers from one corpus to another** (e.g. the FAA reorganizes Part 91 enforcement into a new TSA corpus). Aliases handle within-corpus moves but not cross-corpus moves.
6. **A scenario fixture file imports a constant identifier from `@ab/sources`** but the registry entry was retired between the fixture's write and its run. The fixture compiles; the runtime resolves to retired. What does the test see?
7. **A lesson is co-edited by two agents in parallel branches** and both agents auto-stamp the same unpinned reference to different editions (A pulls main with `accepted` 2026; B pulls main with `accepted` 2027 because ingestion landed in between). Merge conflict on the pin.
8. **The renderer is asked to render `@text` on a section that was redacted in the indexed tier.** What does it substitute?
9. **A `+ack` annotation references a superseder that no longer exists in the registry** (the superseder was retired and superseded itself). Cascade resolution required; not specified.
10. **A lesson rendered to a snapshot (cached PDF, archive.org)** weeks/months/years later. Which edition does the snapshot reference? The pin tells you, but the snapshot's renderer was the version at snapshot-time -- maybe with different token-substitution rules. Reproducibility not addressed.
11. **An author wants to cite "the section that was numbered 91.103.b.2 in 2025 but is now 91.103.c.2 in 2027"** -- a deliberately-historical reference. The ack-historical mechanism handles "I want to cite the original," but doesn't have a clean way to pin "the original numbering as it was in 2025."
12. **A registry alias chain crosses a corpus rename.** Aliases are within-corpus. Cross-corpus aliases would require schema extension; not addressed.
13. **An author writes `[*emphasized*](regs:cfr-14.91.103@2026)` with Markdown formatting in the link text.** Does the validator check link-text-non-emptiness on the rendered text or the source text? Markdown emphasis inside link text is allowed but the §1.5 ERROR rule "link text empty" doesn't define "empty" against marked-up text.
14. **Whitespace-only registry diff.** Ingestion produces text differing only in trailing whitespace or line endings (CRLF vs LF). Hash-compare in §5 says "needs human review." Author opens the diff -> no semantic change. Repeated for every section every year. Ingestion should normalize before hashing.
15. **A regulation that's `[Reserved]`-then-amended-back.** Section was reserved in 2026, gets re-occupied in 2030 with new text. Edition history has gaps, then content. Pin to 2026 (reserved) is meaningful in some readings but useless in most. Lesson-author intent unclear from pin alone.
16. **Authoring a lesson that's deliberately about an obsolete regulation** -- a historical-context lesson. The whole lesson is in `+historical` mode. Authoring overhead is high (every link gets `+historical`). Frontmatter-level mode would be cleaner.
17. **A reference inside an HTML embed inside a Markdown lesson** (`<details><summary>...</summary>[regs:...](...)</details>`). Different parser path; identifier extraction may miss it.
18. **A reference in a Svelte component file that renders as part of a lesson** (lesson body imports `<NightIFRChecklistDisplay />`). Identifiers in TypeScript, not Markdown. §9's row "scenario fixture file" partially addresses but doesn't extend to live components.
19. **Identifier collision across corpora.** Could `regs:cfr-14.91.103` and some future `regs:cfr-14.91.103` mean the same thing in two registry shards? Globally-unique-by-construction is asserted but not enforced.
20. **An author writes `@cite` in a context where the registry entry's `canonical_title` is null** (e.g. a Reserved section). The token substitutes... what? Empty string? "Reserved"? Error?
21. **The `airboss-ref:`-style resolver service URL** (suggested in §9) is a domain we don't yet own, an HTTP service that doesn't yet exist, and a piece of infrastructure with no work package. Listing it as a fall-through for external viewers presupposes infrastructure that's unbuilt.
22. **Validator behavior for `regs:` corpus when the corpus's resolver hasn't been registered yet.** Per §2.2, each corpus registers a resolver. If a lesson references `regs:` and the `regs` resolver hasn't been registered yet (Phase 4 hasn't landed), what does the validator do?
23. **Versioning workflow + ack interaction.** Diff job advances `regs:cfr-14.91.103` from `@2026` to `@2027`. The lesson has a `+ack` against the Walker letter. Does the auto-advance preserve the ack? Strip it? Re-validate it against the new pin?
24. **Internal cross-referencing (lesson A references lesson B which references regulation X).** Transitively, lesson A is about regulation X. Should the registry surface this in `findLessonsCitingFromLessons`? Currently the function is direct-reference only.
25. **Authoring a lesson that's deliberately ambiguous** about which edition a claim applies to ("the FAA has historically required X"). The mandatory-pinning rule (§1.3) forces a pin; some authoring intentions don't fit.
26. **A reviewer comment on the registry promotion path.** §2.4 promotes `pending` -> `accepted` "An engineer reviews each per-corpus batch." Who's the engineer? Joshua? An LLM? Both? Recovery if a wrong promotion is caught later?
27. **The `last_amended_date` for a section that was created (not amended) in this edition.** Is its last-amended-date its publication date? Or null? Schema says "non-null; defaults to published_date" but that's a writer's default; semantically a never-amended section has no last-amended event.

## Recommendations

In priority order. Each is concrete enough that a change to the ADR closes it.

**Priority 1 -- before approval:**

1. **Resolve C1 (colon collision).** Pick a separator strategy that doesn't collide with URI-scheme parsing, OR write a §1.1 subsection acknowledging the trade-off and committing the workarounds. The current silence on this is the biggest hidden cost.
2. **Resolve C3 (ack syntax).** Move acknowledgments out of the link `title` attribute. Frontmatter or comment-block annotation. The current shape will not survive contact with editors.
3. **Resolve C2 (auto-stamp). At minimum:** specify CI behavior (no writes); specify the override marker; specify staleness rule; specify multi-machine reconciliation.
4. **Resolve C4 (phasing).** Move CFR ingestion before the renderer in §8, OR specify validator's "registry-not-yet-populated" behavior.
5. **Rewrite C5 (acceptance criteria) as concrete machine-checkable items.** Match ADR 018's pattern.

**Priority 2 -- before Phase 1 builds:**

6. **Resolve S1 (publish gate).** Define "published," specify the gate's tooling, list the WP that builds it.
7. **Resolve S5 + N7 (lifecycle states + aliases naming).** State-machine diagram in §2.4 with all states declared in §2.1; rename the colliding `aliases` fields.
8. **Resolve S6 (`unknown:` mechanics).** Either it's a corpus with a real resolver, or it's a magic prefix; not both.
9. **Resolve S7 (Phase 9 migration spec).** Either flesh out the migration rules or move it to a follow-on WP and cite the WP.
10. **Resolve N4, N6 (schema interface drift).** `Edition.aliases.to` typed as `string | string[]`; clarify `last_amended_date` cardinality at section level vs edition level.
11. **Resolve S9 (§13 changelog errors).** Three rows are wrong; correct them.
12. **Tighten S10 (deferred triggers).** Replace "when needed" / "when learner-accounts ship" with concrete dates or events.

**Priority 3 -- before Phase 3 builds:**

13. **Resolve S3 (render orchestrator).** Expose batch-resolve from `@ab/sources/render`; require all surfaces to consume it.
14. **Resolve S4 (render-mode token interactions).** Specify TTS behavior for `@text`/`@quote`; specify tooltip behavior for author-overridden tokens; specify transclusion edition resolution.
15. **Address S8 (cross-reference staleness).** At minimum, document that pair-wise staleness exists and isn't currently caught; add to the deferral table with a trigger.
16. **Walk every "Cases the ADR still doesn't address" item.** For each, pick: resolve in ADR, plan in WP with trigger, defer with concrete trigger, drop. None survive as TBD.

**Priority 4 -- nice-to-have:**

17. **Address N1-N15.** Style and consistency.
18. **Add a §15 "Worked failure example."** §12 walks the happy path. Add: lesson with stale pin + ack against superseded letter + rendered to PDF + pulled into a quiz card. Forces the supersession-chain + ack-resolution + multi-renderer cases to surface.

---

The v2 is approvable with the four Priority 1 items resolved. The user's worry -- "we don't want to redesign and rebuild references because we didn't think something through" -- is reasonable to invoke against (C1) and (C3) specifically. Both are syntax-level decisions that, once 200 lessons reference identifiers under them, are extremely expensive to revisit. Both are still wrong as written and should be fixed before any lesson commits to either.

The other three Priority 1 items (auto-stamp semantics, phasing, acceptance criteria) are mechanical -- they're the kind of thing where being explicit costs a few paragraphs and saves a Phase-1 rework. Worth doing. Then approve.
