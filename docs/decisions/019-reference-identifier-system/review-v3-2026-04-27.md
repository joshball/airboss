# Review of ADR 019 v3 -- Reference Identifier System

## Summary

**Verdict: approve with revisions.** v3 is a real rewrite, not a v2 with paint, and the load-bearing decisions (real URI scheme, mandatory pinning with `--fix` semantics, frontmatter acks, five tiers with explicit dependency contracts, lifecycle state machine, CFR ingestion before renderer) are the right calls. The four-move spine is intact and clearer. The §10 deferrals + revisit.md sidecar finally make the deferrals discipline real instead of decorative.

That said, three things still need fixing before approval, and one of them is structural rather than cosmetic. (1) The `airboss-ref:` scheme is *almost* right but the ADR doesn't pin the URL form (path-rootless vs. path-absolute vs. authority-based) and the WHATWG URL parser silently produces three different `pathname` values for the three forms a copy-paste round-trip can produce. (2) The validator's "Cross-section" alias rule (§1.5) is ERROR with "auto-resolve refused" but §6.1's `cross-section` AliasEntry kind requires the resolver to actually resolve the chain - those two rules contradict. (3) §3.4 acknowledgments are bound by `target` only, which collapses the case where one lesson references the same target multiple times in materially different contexts. None of these will sink the ADR; all are within-§ fixes; all should land before the first lesson commits to v3.

There are also a meaningful number of medium-severity gaps the §14 changelog doesn't acknowledge: the `--fix` behavior in CI vs. dev still lets sequential CI runs on a stale branch silently age, the lifecycle state machine doesn't model the "concurrent ingestion + un-promotion" race v2 §8 §S9 raised, the publish gate handwave doesn't define how feature-flagged or pre-merge preview deploys interact with "publish = merge to main," and the `unknown:` magic-prefix vs. corpus distinction is *closer* but the validator's parsing rule for `airboss-ref:unknown/<anything>` isn't actually written down. None block approval; all are worth a paragraph each.

The revisit.md sidecar is good. R6 (resolver service) and R10 (registry-promotion delegation) are well-shaped triggers; R2 (audio surface) and R5 (hangar UI) are real but the trigger language ("when X ships") loses force if the surfaces never ship. R13 (academic citation) is genuinely the kind of "if real need arises" item that v3 was supposed to drop, not defer with a soft trigger; recommend dropping outright. R20 (grep affordance) is so resolved by the design itself ("paths contain section numbers") that it doesn't belong on a revisit list.

## Critical concerns (new in round 3)

### C1. The URI form is underspecified; the WHATWG URL parser produces three different `pathname` values for what an author thinks is one identifier

§1.1 specifies `airboss-ref:<corpus>/<locator>?at=<edition>` and asserts the scheme survives every URI library. That's almost true. But the WHATWG URL Standard (which Node, browsers, fetch, all major JS URL libraries use) treats `airboss-ref:` as a non-special scheme, which means the URL constructor accepts three syntactic variants:

```text
airboss-ref:regs/cfr-14/91/103?at=2026     pathname = "regs/cfr-14/91/103"     (path-rootless)
airboss-ref:/regs/cfr-14/91/103?at=2026    pathname = "/regs/cfr-14/91/103"    (path-absolute)
airboss-ref://regs/cfr-14/91/103?at=2026   pathname = "/cfr-14/91/103"         (with authority)
                                                                                hostname = "regs"
```

All three are valid RFC 3986 URIs. All three round-trip through the URL constructor. Only the first matches the ADR's specification. An author copy-pasting from a context that normalizes URLs (some Markdown editors auto-add `//`, some browsers normalize path-rootless to path-absolute) will produce a syntactically-valid identifier with a different parse tree. The validator's `parseLocator` will see `corpus = ''` (when authority is the corpus) or `corpus = '/regs'` (when path is rooted), and the lesson silently fails resolution at render time even though `bun run check` passes.

A second related issue: the WHATWG URL constructor silently strips trailing whitespace and normalizes some characters. `airboss-ref:regs/cfr-14/91/103?at=2026 ` (trailing space, common from copy-paste) parses identically to the cleaned form. The author can't see the difference; the validator can't see the difference; the parser silently accepts both.

A third related issue: relative URL resolution. `new URL('foo', 'airboss-ref:regs/cfr-14/91/103?at=2026')` throws `Invalid URL`. So the renderer cannot use the WHATWG URL machinery for any link rewriting (transclusion, deep-link composition, etc.). It must operate on the raw string. The ADR doesn't say so.

**Why it matters:** the round-2 critique was "the colon collides with URI parsers." v3 fixes that. But it doesn't pin the *URI form*. Five years from now, an author copy-pastes from a renderer that emitted `airboss-ref://regs/...` (because some downstream transform normalized it), the lesson commits, the validator passes, the lesson silently breaks. This is exactly the corner the user explicitly worried about.

**What to do:** add §1.1.1 specifying:

- The canonical form is path-rootless (`airboss-ref:regs/cfr-14/91/103?at=2026`). The validator REJECTS any other form.
- The validator normalizes whitespace before parsing (trim leading/trailing on the raw URL string).
- Relative URL resolution is not supported; the renderer treats identifiers as opaque strings except for `parseLocator`.

Three sentences. Closes the corner.

### C2. The `unknown:` magic-prefix parsing rule is not written down, and §1.5's first row puts the validator in a contradiction

v3 §1.7 says `unknown:` is a magic prefix recognized by the validator, NOT a corpus with a `CorpusResolver`. Good. But:

- §1.1 says identifiers parse as `<corpus>/<locator>?at=<edition>`.
- §1.5's first row: "Identifier must parse (corpus exists, syntax valid for corpus): ERROR."

If the validator uses `corpus exists per §1.2 enumeration` -> `unknown` is not enumerated -> first-row ERROR is the only error emitted. But §1.7 says `unknown:` is a magic prefix that emits ERROR for a different reason ("ERROR. The lesson cannot merge"). Both rules fire on the same input. Which one wins? Are both ERRORs emitted, or just one? The author sees one error message that says "magic prefix" or two error messages saying both?

A second contradiction: §1.5 row 1 says "ERROR" for parse failure. But §1.5 row 2 says "Identifier resolves to an `accepted` or `superseded` registry entry: ERROR." For an `unknown:` identifier, both rows fire. Confusing if the author is reading the validator output trying to figure out what's wrong.

A third specification gap: per §1.7, `unknown:descriptive-string` parses (it's syntactically `<corpus>/<locator>` with `corpus = unknown` and `locator = descriptive-string`). Per §1.5 row 1, "corpus exists" is required. Is `unknown` in the corpus enumeration of §1.2? It isn't listed. So row-1 fires by enumeration check. But §1.7 explicitly carves out `unknown` as a special case.

**Why it matters:** the validator's job is to give clear errors. Two overlapping errors on the same input produce confusing messages. Worse, the author who writes `unknown:foo` probably did so deliberately during transition; the validator should say "this is an unresolved transitional reference; either ingest the corpus or replace with a real identifier" - not "corpus does not exist" plus "identifier does not resolve" plus "magic-prefix ERROR."

**What to do:** explicitly state in §1.5:

- Row 1 ("identifier must parse") fires only when the literal string fails URL parsing (per C1 form rules) OR has invalid characters per the corpus's resolver. For `unknown:`, the magic-prefix rule applies and row 1 does NOT fire.
- Add a row 0 (highest priority): "Identifier uses `unknown` magic prefix: ERROR with message 'Transitional reference; cannot publish. Replace with a real identifier or wait for ingestion of the relevant corpus.'"
- The validator emits exactly one error per identifier, not multiple.

### C3. §1.5's "Cross-section move: ERROR (auto-resolve refused)" contradicts §6.1's `cross-section` AliasEntry kind, which has a working `to` resolver

§1.5 row "Cross-section move" assigns ERROR with "auto-resolve refused; author must re-pin." §6.1 defines `cross-section` as one of the AliasEntry kinds; the resolver walks the chain and applies aliases in order. If the resolver applies a `cross-section` alias and emits ERROR + refuses to resolve, the lesson is permanently broken until the author edits it - which is correct behavior, but then §6.1's "Aliases chain. Resolver walks from the lesson's pinned edition forward, applying aliases in order" is wrong for `cross-section`: the chain stops there.

Two scenarios:

1. Lesson pinned to 2026; in 2027 the section moves cross-section. Validator emits ERROR. Lesson is broken. Author re-pins manually to the new identifier. Now the lesson is pinned to 2027.
2. Lesson pinned to 2026; in 2027 the section moves cross-section; in 2028 it moves again (different cross-section, same content). The chain has two `cross-section` hops. The first hop ERRORs; the lesson never reaches the second.

§6.1 says "Chain depth is unbounded; lessons N years old still resolve via traversal." But for cross-section, traversal is refused at the first hop. The two §s contradict.

**Why it matters:** the resolver has to know whether a `cross-section` chain is an error-on-first-hop or an error-on-every-hop or an error-only-at-resolution-time. §1.5 and §6.1 implement different rules.

**What to do:** decide what `cross-section` means and write it down once:

- Option A (my recommendation): `cross-section` is a hard stop. The resolver does NOT walk past a `cross-section` alias. The lesson's reference is broken; author must manually re-pin. §6.1 should say "chain walks until it hits a `cross-section` alias, then stops with ERROR."
- Option B: `cross-section` is a soft stop. The resolver walks past it but emits ERROR every time. Lessons resolve to the latest target but the build fails. Author manually re-pins to suppress.
- Option C: `cross-section` only ERRORs the first time it's encountered (to alert the author); future lessons authored after the alias is in place use the new identifier directly. This is the "transitive resolution" model and requires §6.1 to specify "the first alias of each kind ERRORs; subsequent aliases of the same kind are silent."

Pick one. The current text reads as Option A in §1.5 and Option B in §6.1.

### C4. Acknowledgments are bound by `target` only, which loses information when one lesson cites the same target multiple times in different contexts

§3.4 acknowledgments key on `target: airboss-ref:interp/chief-counsel/mangiamele-2009`. The lesson body can reference Mangiamele in two materially-different contexts:

```markdown
The Mangiamele letter (2009) [...common-purpose discussion...] established the
limit on cost-sharing under §61.113.

Later, in the context of compensation broadly, the [Mangiamele letter](airboss-ref:interp/chief-counsel/mangiamele-2009)
also addresses [...different facet...].

In yet another reference, the Mangiamele analysis of "common purpose" was
cited approvingly in [...different context...].
```

If the lesson has one ack with `target: airboss-ref:interp/chief-counsel/mangiamele-2009`, that ack applies to *all three* references. But the author's intent might be:

- For reference 1, "original-intact" (the cost-sharing analysis is unchanged by Smith-2027).
- For reference 2, "scope-narrowed" (Smith-2027 narrowed the broader compensation discussion).
- For reference 3, "historical only" (the common-purpose discussion is now superseded by a different doctrine).

§3.4's data model can't express that. The frontmatter binding is per-target, not per-reference.

A second related issue: a lesson with 20 acknowledgments. The frontmatter becomes a wall of YAML (20 entries × 5 fields = 100 lines). Authoring overhead is real. The link-title syntax v2 tried (and v3 rejected) had the locality property: acknowledgment is at the link site. v3 chose frontmatter for editor-fidelity reasons; the trade-off is locality. Worth acknowledging the trade-off.

A third related issue: when a lesson is split into two lessons, the acks in the original frontmatter must be split between the two new lesson files. This is a textual operation but requires the author to know which ref goes to which sub-lesson. Easy to get wrong; validator catches via "orphan ack target" but the *positive* case (every body-ref has its right ack) isn't caught.

**Why it matters:** v3 picked frontmatter for editor-fidelity. The cost is per-link locality. The cost shows up specifically when a lesson references the same target multiple times for different reasons. This is rare but real (Mangiamele has multiple facets; Walker likewise; Garvey-Lobeiko has multiple holdings).

**What to do:** pick one:

- **Add per-link binding via Markdown link reference labels.** `[Mangiamele letter][mangiamele-cost-sharing]` and frontmatter has `acknowledgments: - { id: mangiamele-cost-sharing, target: ..., reason: ..., context: cost-sharing }`. The reference label binds the ack to the specific link. Editors handle reference-style links robustly (they're standard CommonMark).
- **Accept the limitation.** Document in §3.4: "An ack applies to all references in the lesson body to the named target. If the lesson references the same target in materially different contexts, split the lesson or use distinct identifiers (e.g., the document level vs. specific section)." This is the cheap answer; it's only wrong for the rare case.
- **Allow multiple ack rows with the same target, distinguished by `reason` slug.** When the body cites the target, the validator checks that *some* ack with that target is present; the renderer chooses which to display by checking the surrounding prose for matching reason-slug context. (Brittle. Don't recommend.)

Recommend reference-label binding. It's standard Markdown, survives editor round-trips, gives per-link locality, and the validator can enforce that every label is bound.

## Significant concerns (new in round 3)

### S1. The `--fix` semantics resolve C2 from round 2 but introduce a "stale CI on a dormant branch" gap

§1.3's failure-mode table is largely good. One scenario isn't captured:

Author opens PR in 2026. PR sits unreviewed for 6 months. In 2027, ingestion lands the new edition. CI re-runs on the PR branch (because main moved). The validator sees the PR's pins are still `?at=2026`; per §1.5, "Pinned edition is older than current `accepted` by > 1 edition: WARNING." It doesn't ERROR (still within 1 edition). PR merges; lesson ships with 2026 pins; the diff job picks it up next cycle.

That's actually fine for the 1-edition case. But:

Author opens PR in 2026. PR sits unreviewed for 18 months. CI re-runs on the PR branch (main has moved through 2027 and 2028). Pins are still `?at=2026`; that's 2 editions stale; per §1.5, this should be a WARNING (`> 1 edition`). It is. So the PR ships with a warning visible in CI but doesn't block. The lesson lands with stale pins; the diff job picks it up next cycle.

This is correct behavior for both cases. But the §1.3 table doesn't say what happens when CI re-runs on a stale branch (the PR was opened before the registry advanced). Worth a row.

**What to do:** add a row to §1.3:

| CI re-run on a stale PR branch | `bun run check` runs without `--fix`; emits WARNING for any pin `> 1 edition` stale; doesn't block; PR can still merge. Author re-runs `--fix` locally if they want to advance pins; otherwise the diff job catches it next cycle. |

### S2. `@text` in TTS for a long lesson reads every regulation aloud; the design accepts this but doesn't warn authors

§3.1's TTS row: "Author's choice of token determines length: `@cite` is read fully; `@text`/`@quote` reads the regulation aloud (intentional - if you wanted shorter, use `@cite`)."

That's a design choice, not a bug. But the lesson author writing for the web context, then having the lesson rendered for audio, doesn't necessarily anticipate that a 50-cite lesson with `@text` everywhere becomes 30 minutes of regulation-text audio.

The validator could surface a WARNING: "This lesson uses `@text`/`@quote` on N references. In TTS rendering, this will read every regulation aloud. Consider `@cite` for the audio rendering surface." Author either accepts (intentional) or revises.

**What to do:** add a NOTICE-tier rule:

| Lesson uses `@text` or `@quote` on > 5 references and is in a corpus that ships an audio surface | NOTICE | Surface the audio-rendering implication so author can choose `@cite` if not intended |

Or: add a per-render-mode token override syntax (e.g. `@cite[audio]:@text[web]`) and let authors specify mode-specific tokens. Probably more complexity than warranted; the NOTICE is enough.

### S3. The publish gate is "merge to main" but the actual deploy story has more layers

§1.6: "Publish means merge to main. No separate publish gate tooling. The validator's ERROR-tier rules ARE the publish gate."

This works for the simple case. But airboss has at least three deploy contexts:

- **Pre-merge previews.** A PR's branch deploys to a preview URL. Lessons render. If a lesson has `unknown:` references, the preview shows them with the [unresolved] badge. This is intentional during development. But the preview is *visible* to anyone with the URL. Is that "published"?
- **Feature-flagged lessons.** A lesson lands on main but is gated behind a feature flag (e.g., visible only to seed users). Is that "published"?
- **Deploy environments (dev/staging/prod).** Same git ref, different runtime config. Lesson with stale-but-valid pins ships to all three. Is each "published"?

The §1.6 rule treats all of these the same way: merge-to-main = published; CI runs at merge time; ERROR-tier rules enforce. That's the simple answer and it might be the right one. But the user's "the hard cases surfaced" ask should make us check.

**What to do:** add a sentence to §1.6:

> "Publish" in this ADR means "merge to main." Pre-merge previews, feature-flagged content, and deploy environments are all downstream consumers of merged content; each may have its own additional gates (preview rendering may show `[unresolved]` badges; feature flags may gate visibility), but none is a separate validation gate.

This collapses the corner-cases into the same rule and explicitly says "no, we don't validate separately at deploy time."

### S4. Cross-corpus supersession in §6.2 is allowed; the renderer behavior is not specified

§6.2: "Cross-corpus supersession is supported. The pointer crosses corpus boundaries."

Scenario: a lesson references `airboss-ref:regs/cfr-14/91/103?at=2026`. In 2030, Part 91 is reorganized; the section moves to a new TSA corpus as `airboss-ref:regs/tsa-49/some-section?at=2030`. The supersession pointer crosses corpora.

The renderer batch-resolves identifiers via `@ab/sources/render`. It walks the supersession chain. The current lesson's reference resolves... where? To the original 2026 identifier (which still exists in the registry as `superseded`)? To the successor in the new corpus (a different render context)? Both?

§6.3 says "the renderer adds a 'via [chain]' annotation." So the renderer presumably renders the original reference plus an annotation pointing at the successor. But the cross-corpus successor has a different `canonical_short`, possibly a different `formatCitation` style. The annotation's text (e.g., "(via §[some-tsa-section])") is unspecified.

**What to do:** add a sentence to §6.2:

> When the supersession chain crosses corpora, the renderer displays the original reference's `canonical_short` (with annotation indicating cross-corpus supersession) plus a "see also" link to the successor's identifier in the new corpus. The successor's per-corpus resolver provides the citation format for the annotation.

Or specify a different rule. The point is the renderer needs a contract.

### S5. The `@ab/sources/render` API is consumed by 7 surfaces with very different runtime profiles

§2.5's table lists 7 surfaces all consuming `@ab/sources/render`. This is the right architecture but the runtime constraints differ:

- **Lesson page (SvelteKit SSR + hydration):** has Postgres connection, has SvelteKit, has all of `@ab/*`. Standard.
- **CLI PDF export:** likely has Postgres connection, no SvelteKit, runs once.
- **Share-card preview generator:** runs in Lambda or similar. May or may not have Postgres connection. Tight memory and time limits.
- **Search-result snippet:** runs inline with search service, may be a separate process.
- **RSS feed generator:** scheduled job, may run on different infra than web app.
- **CLI bibliography export:** local-machine tool, possibly running offline.
- **RAG citation snippet builder:** runs in vector-DB pipeline, possibly different language (Python).

`@ab/sources/render` is TypeScript. The Python RAG pipeline can't import it directly. The Lambda image builder must bundle it. The CLI tools must have a way to point at the registry without running the SvelteKit app.

§2.5 says "All surfaces consume the same `@ab/sources/render` API. No per-surface batch-resolve reimplementation." That's the goal. But the goal isn't realistic for surfaces that aren't TypeScript / aren't online / don't have Postgres.

**What to do:** §2.5 should distinguish two layers:

1. **The pure TypeScript layer** (`extractIdentifiers`, `substituteTokens`) - works anywhere, no async, no Postgres. Consumed by all TS surfaces.
2. **The data-fetching layer** (`batchResolve`) - requires registry access. Has at least two implementations:
   - In-process (TypeScript, has Postgres): used by SvelteKit, CLI tools.
   - Snapshot-based: a JSON dump of the registry's static + indexed tier is generated at deploy time and consumed by surfaces that can't reach Postgres (Lambda, RAG pipeline, offline tools).

For non-TypeScript consumers (RAG / Python), the snapshot is the contract. The static registry is a JSON dump; resolving is a hash-map lookup; no `@ab/sources/render` dependency.

This is a meaningful architectural detail. Worth a sentence.

### S6. The "atomic batch promotion" rule in §2.4 doesn't define what a batch is

§2.4: "Entries are promoted in batches (e.g. 'all of Part 91 sections 1-50') or not at all. Half-promoted batches are forbidden. The reviewer signs off on the full batch."

But:

- What's the size of a batch? Reviewer's discretion? Per-corpus convention? Engineer-author decision?
- Can a batch span multiple corpora? (Probably no, but not stated.)
- Can a batch be a single section? (Probably yes for corrections, but not stated.)
- Are batches recorded as first-class objects (each batch has an ID, a reviewer, a date)?
- What's the recovery if a batch is promoted, then a section in it is later found wrong - does the whole batch un-promote, or just that section?

**What to do:** §2.4 should specify:

- A batch is defined by the ingestion pipeline's run unit (e.g., one CFR Part, one AC Change document, one AIM section).
- Batches are recorded in a registry-internal table (`promotion_batches`) with reviewer ID, date, scope.
- Recovery: a single section in a batch can transition to `retired` independently; the rest of the batch stays `accepted`.

### S7. `historical_lens: true` lesson-level shortcut interacts oddly with per-target acks

§3.4: `historical_lens: true` "every reference in the body is treated as `+historical` without per-link annotation."

But the lesson can also have per-target `acknowledgments:` entries. What happens when a lesson has BOTH `historical_lens: true` AND a per-target ack for a specific reference?

Two interpretations:

- (A) `historical_lens: true` overrides everything; per-target acks are ignored.
- (B) `historical_lens: true` is a default; per-target acks override specific references (e.g., "this whole lesson is historical EXCEPT for the Walker reference, which is still applicable").

§3.4 doesn't say.

**What to do:** state explicitly. (B) is more useful (it lets a lesson be mostly-historical but call out one current reference). But (A) is simpler. Either is fine; pick.

## Minor concerns / nits (new in round 3)

### N1. §3.1 "Plain-text export" appends "resolver service URL" - resolver service is dropped

§3.1's "Plain-text export" row says "**resolver service URL** appended as `(<URL>)`." But the resolver service is dropped from v3 scope (§10). Should be "per-corpus live URL" or "the URL returned by `getLiveUrl(id, edition)`."

### N2. §3.1's "Slack / iMessage unfurl" assumes airboss is in the path

The unfurl agent fetches a URL and extracts metadata. For an `airboss-ref:` URL pasted into Slack, the unfurl agent has no handler - it falls back to displaying the raw URI string. §3.1 says "Title from `@cite`; description from lesson title" but that requires the lesson page on `study.airboss.dev` to handle a redirect from an `airboss-ref:` URL. Not specified.

### N3. §1.5 row "Pinned edition is `pending`, `draft`, or `retired`" lumps 3 states with different reasons

A `pending` ref means "ingested but not reviewed yet" (transient); `draft` means "corpus author still writing" (also transient but earlier); `retired` means "was correct, no longer current" (intentional). Treating all three as ERROR is fine but the error message should distinguish ("entry is pending review", "entry is in draft", "entry is retired").

### N4. §2.1 `SourceLifecycle` declares 5 states; §2.4 transition diagram only shows 4

```text
draft → pending → accepted → retired
                       ↓
                  superseded
```

There's no inbound arrow to `superseded`. Per §2.4, supersession "via supersession relationship (creating a new entry with `supersedes` pointing back) or manual mark." So `accepted → superseded` is via two paths. The diagram should show both, or §2.4 should say "any state can transition to `superseded`."

### N5. §2.3 query API mixes sync and async without convention

```typescript
resolveIdentifier(id: SourceId): SourceEntry | null;          // sync
walkSupersessionChain(id: SourceId): SourceEntry[];           // sync
findLessonsCitingEntry(id: SourceId): LessonId[];             // sync
findLessonsTransitivelyCitingEntry(id: SourceId): LessonId[]; // sync
getEditions(id: SourceId): Promise<Edition[]>;                // async
isPinStale(id: SourceId, pin: EditionId): Promise<boolean>;   // async
```

Sync vs. async is determined by whether the call hits the indexed tier. Worth explicit comment: "Static-tier queries are sync; indexed-tier queries are async."

### N6. §3.4 `reason` slug rules say `[a-z0-9-]` max 64 chars but examples use `original-intact` which is 15 chars

Fine, just observing the recommended-16-32 range conflicts with shorter natural examples. Either widen the recommendation or use longer examples ("scope-narrowed-but-91-103-claim-intact" from v2's example was 39 chars and natural).

### N7. §13's worked failure example uses `renderer_version` frontmatter that's not formally introduced

§3.4 lists the frontmatter fields. `renderer_version` isn't there. §13 says it's "added at snapshot time" - i.e., not author-authored. Should §3.4 mention auto-generated fields, or §3 list `renderer_version` explicitly with a note "set by the snapshot pipeline, not by authors"?

### N8. §2.6 says "Lessons should not be authored for a corpus until that corpus's bulk ingestion has landed and its registry entries are `accepted`"

Reasonable rule. But the validator doesn't enforce it - it enforces "the specific identifier resolves." A lesson author can write `airboss-ref:aim/5-1-7?at=2026-09` even if the AIM ingestion has only loaded `pending` entries. The lesson's reference fails resolution; build fails; author waits. That's the same outcome as the rule; the rule is descriptive not prescriptive. Worth saying: "this is enforced by the validator's resolution rule, not by a separate gate."

### N9. §1.3 row "Author wants explicit unpinned" cites `?at=unpinned` as WARNING

This is the right rule. But "unpinned" as a magic value in the `at=` query parameter slot isn't the same as "no `at=` at all." The validator must distinguish:

- No `at=` and no slug-encoded edition -> ERROR (per §1.6 publish gate).
- `?at=unpinned` literal -> WARNING (per §1.3 row).
- Slug-encoded edition (e.g., `8083-25C`) -> OK (no `at=` needed).

Three states, one parameter slot. Worth one sentence ensuring the validator knows the difference.

### N10. §11 acceptance criteria item "§10 has zero rows with weaker triggers than 'when X ships' or 'when Y is requested' or 'DROPPED'"

Item is sound. But phrasing "zero rows weaker than..." defines the threshold by example. A clearer rule: "Every row in §10 maps to either (a) a revisit.md item with a concrete trigger event, or (b) DROPPED with a reason." This is checkable by greppable rule.

### N11. §3.4 frontmatter example shows `note` as free-form prose

`note: "Smith narrows but does not overturn the cost-sharing analysis cited here."` - rendered as part of the output. Good. But §3.1's render-mode table doesn't say where the `note` appears in each render mode. Tooltip? Footnote? Inline? Spec it.

### N12. §1.4 `@list` token: "single-reference `@list` produces the canonical_short of that one entry (a list of one)"

Works. But "list of one" rendered means... just the canonical_short, no separators. The author may not expect that. Either say "for single-reference `@list`, the output is identical to `@short`" or use `@short` always and remove the "list of one" carve-out.

### N13. §3.3 says "All identifier constants imported by Svelte components rendered as part of the lesson"

How does the parser know which Svelte components are "rendered as part of the lesson"? The lesson Markdown imports a component; the component imports an identifier. The walk has to follow Svelte component imports through TypeScript modules. That's a real graph walk. Worth one paragraph on the parser's strategy (AST walk? bundler-assisted? naive grep of imported files?).

### N14. §6.1 alias chain "Gaps (an edition was skipped during ingestion) are an ERROR"

A gap is detected by the resolver when walking the chain. But the validator runs at lesson-author time, not at ingestion time. Authors don't know if there's a gap until they write a reference and run check. Should ingestion itself detect gaps and report them as ingestion-level errors before any lesson is authored? Probably yes. Worth saying.

### N15. §10 row "Hangar UI for non-engineer-edited registry entries" trigger is "when `apps/hangar/` ships content authoring"

`apps/hangar/` is dormant per the project's recent post-pivot state (per the CLAUDE.md note in the user's project context). The trigger fires when a dormant app ships - maybe never, maybe in 2 years. revisit.md R5 has the same trigger. Acceptable but worth noting that "hangar surfaces are reawakened by separate trigger" per ADR 017.

## Cases v3 still doesn't address

Listing scenarios v3 can't handle gracefully or doesn't specify clearly:

1. **A lesson references the same target multiple times in materially different contexts** (see C4). Frontmatter ack is per-target, not per-reference. Author can't distinguish.
2. **A `@text` token rendered into TTS for a regulation-heavy lesson.** Becomes 30 minutes of regulation-text audio. Design accepts; no warning.
3. **A reserved section's `canonical_title`.** Round-2 had it as "Reserved"; v3 doesn't say. `@cite` substitution undefined.
4. **A section created in an edition (not amended).** `last_amended_date` cardinality undefined per schema.
5. **A future-pinned reference that the author meant as speculative.** ERROR is correct but no recovery path; author either pins to a real edition or loops forever.
6. **A lesson rendered in a non-airboss CommonMark viewer.** The `airboss-ref:` URL is still parseable as a URI by the viewer's URL handler, but no scheme is registered. Result: viewer renders it as text or a broken link. v3 dropped the resolver service; the fall-back is "live URL" but the renderer that produces a live URL is *the airboss renderer.* External viewers see `airboss-ref:regs/cfr-14/91/103?at=2026` as text. v3 doesn't say what to do.
7. **An identifier with embedded `?` in a slug.** Unlikely (no FAA corpus does this) but possible for foreign reg-set names. The query parameter parse rule assumes the first `?` is the boundary. If a slug contains `?` (URL-encoded as `%3F`), the locator includes literal `%3F`; the parser must round-trip the encoding. Not specified.
8. **Round-trip through email or chat.** A lesson body is copied into an email or Slack message. The `airboss-ref:` URL is preserved (it's a valid URL scheme). The recipient pastes it back. Now it's text in a different lesson. v3 says the validator only validates lessons; the email/chat is fine. But the round-trip identity isn't tested.
9. **The renderer output for a token that doesn't apply to the corpus.** `@last-amended` for a section that has no last-amended date (e.g., a newly-created section). What does it render? Empty? `null`?
10. **A lesson's frontmatter `acknowledgments` entry references an `unknown:` target.** Validator emits ERROR for the body reference; what does the ack with `unknown:` target do? Orphan WARNING per §3.4? Or also ERROR?
11. **A lesson is moved between sections of the course or between products.** The lesson's identifier references don't change. But the lesson's URL does; the resolver service (if it existed) would need to follow. v3 dropped resolver service; the lesson moves and the references continue to resolve correctly.
12. **An edition is republished with no changes (a "no-op" edition).** Diff job hash-equal; no human review needed. Auto-advance. But the new edition still increments the registry's edition count, which is now slightly out of sync with the FAA's edition count. Worth a row.
13. **A lesson references a regulation that is amended and then *un-amended* (sunset clause + reinstatement).** Edition history has 2026 = original, 2028 = amended, 2030 = back to original. Lessons pinned to 2026 vs. 2030 see the same text. Diff job's hash-compare catches "no change" between 2026 and 2030; auto-advances. Author may not notice the round-trip.
14. **A user-zero (Joshua) lesson edited by Joshua-the-author and Joshua-the-reviewer in different states.** Single-author workflow doesn't have the multi-author race. But the §1.3 multi-machine workflow rules apply. Worth a sentence: single-author workflow is the default; multi-author isn't tested in v3.
15. **An ack-target lesson body removes the reference but the frontmatter ack still points at it.** Validator emits WARNING (orphan). The user's note in the prompt asks "is WARNING enough? Should it be ERROR?" My take: WARNING is right for the case "I'm refactoring; will fix the ack later," ERROR for the case "I deleted the reference and forgot." But the difference is intent. WARNING with a 30-day "stale orphan" promotion-to-ERROR rule (in CI, not in the validator) might be the cleanest answer. Or just ERROR; the cost of an over-flag is tiny.
