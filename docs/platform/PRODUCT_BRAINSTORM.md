# Product Brainstorm

Modes the platform should think in:

- **Pre-flight (mobile)** -- 5-30 min before takeoff, in FBO / cockpit / car. Need: load context, rehearse decisions.
- **Per-flight reflection (mobile)** -- post-flight, on the ramp or at home. Need: capture, debrief.
- **Daily desk** -- 10-30 min, home/office. Need: deliberate practice, study, build habits.
- **Event cram** -- weeks before BFR/IPC/checkride/interview. Need: targeted high-volume prep.
- **Audio/passive** -- driving, walking, exercising. Need: low-effort high-frequency exposure.
- **In-cockpit reference** -- during flight. Need: trustworthy, glanceable, never wrong.
- **Community/social** -- sometimes. Need: presence + accountability without noise.

## A. Pre-flight (the wedge)

| prd:pre:airport-cards  | **Airport Cards**                          | Personalized card for any airport you're going to: pattern quirks, common runway-in-use, taxi traps, hot spots, local frequencies, landing fee, fuel      |

## Strongest threads (my read)

If I had to pick a few that *together* form a coherent product rather than a feature catalog:

1. **prd:pre:route-walkthrough Route Walkthrough + prd:pre:airport-cards Airport Cards + prd:pre:notam-triage NOTAM Triage** -- this is the per-flight pre-brief product. Use weekly+. The wedge.
2. **prd:prof:ten-minute-ticker Ten-Minute Ticker + prd:prof:calibration-tracker Calibration Tracker + prd:ref:skill-heatmap Skill Heatmap** -- this is the daily proficiency product. Habit-forming. The retention engine.
3. **prd:aud:daily-decision Daily Decision (audio) + prd:aud:ntsb-story NTSB Story (audio)** -- the audio surface. Lowest-friction daily exposure. Marketing flywheel even if free.
4. **prd:evt:bfr-sprint/prd:evt:ipc-sprint BFR + IPC Sprint** -- the cram product. Time-bounded, motivated buyers. If anything is going to fund cover-costs, this is it.
5. **prd:ref:per-flight-journal Journal + prd:exp:replay-your-flight Replay-your-own-flight** -- the long-term skin-in-the-game features that make people stick.

These five threads share a single backbone: scenario engine + content library + per-user model. Same architecture, different surfaces.

## Avionics Strategy

### Build strategy, ranked

1. **Schematic sequence trainer (primary bet).** Web-first. Simplified layout mockup with labeled buttons in approximately-real positions. Teaches the *workflow*, not the pixels. Renders in our style. Primitives: PFD tapes, HSI, CDU grid, soft-key row, menu-and-knob patterns. Same primitives serve G1000-style, GTN-style, Avidyne-style variants. Scenario tells the user "load RNAV 24, vectors to TIPRE" -> user taps the path -> wrong paths annotated with why.
2. **Link out to official trainers.** Where pixel fidelity matters (rare), the platform frames the scenario and runs the debrief, user practices in Garmin's free trainer. We provide pedagogy; Garmin provides fidelity. Zero build cost, zero IP exposure.
3. **X-Plane bridge (v2+).** Small plugin exposes state over websocket. Lets a scenario drive the user's own sim if they have one. Mac-native via XPSDK.
4. **Open the primitives eventually.** The schematic-avionics Svelte components are generically useful outside our product. Candidate for open-sourcing on their own later.
5. **MSFS SDK work -- don't.** Wrong sim for a Mac solo dev. Let the WorkingTitle crowd own that space.

### IP read (corrected)

Four categories; only one matters.

| Category    | What it protects                                  | Our exposure if we build a stylized schematic trainer                                                       |
| ----------- | ------------------------------------------------- | ----------------------------------------------------------------------------------------------------------- |
| Copyright   | Specific code, icons, artwork, pixel arrangements | Low if we draw our own SVGs/icons/fonts                                                                     |
| Trademark   | "G1000", "Garmin", logos                          | Low if product is not trademark-named; nominative use ("teaches G1000-style workflow") is industry standard |
| Trade dress | Distinctive overall look/feel                     | **The real axis.** Pixel clones = risk. Stylized similar-position mockups = industry norm                   |
| Patents     | Specific functional inventions                    | Near-zero for UI training material                                                                          |

**Industry precedent.** Sporty's, King Schools, Gleim, ASA, Jeppesen, Max Trescott's *G1000 Glass Cockpit Handbook*, and countless YouTube/CFI materials have shown and taught G1000 workflow openly for 20 years. Garmin has not been litigious about third-party training.

**What would actually draw heat (avoid):**

- Product literally named "G1000 Trainer" or using Garmin marks in the product name.
- Ripped sprites, screenshots, or official icons/fonts.
- Clone so visually identical a customer could believe it's a Garmin product.
- Commercial positioning as a substitute for Garmin's own trainer.

**What's defensible (do this):**

- Generic product name: "integrated flight deck trainer" or similar.
- Everything drawn ourselves -- SVGs, fonts, colors. Visually ours even if positionally similar.
- Functional labels (NRST, DIRECT-TO, PROC, FPL, CLR, ENTER) are standard across avionics families and non-trademarkable.
- Nominative descriptions: "Workflow compatible with Garmin G1000 NXi, Avidyne IFD, and similar integrated flight decks."
- Clearly transformative/educational framing -- scenarios, annotations, "why this button" explanations -- not a pixel twin.
- If Garmin ever objects, change it cheerfully. No stake in specific visuals.

### Net direction

Build the schematic trainer. Primary avionics surface for v1. Design it so primitives can later render "G1000-like," "Avidyne-like," "GTN-like" layouts from the same underlying workflow model -- which is where the real pedagogical value lives anyway (the sequences transfer across boxes; the pixels don't).
