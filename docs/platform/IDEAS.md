# Ideas

Ideas, approaches, and insights that might shape the course or platform. This is the intake funnel -- everything lands here first. Good ideas get promoted to [DESIGN_PRINCIPLES.md](DESIGN_PRINCIPLES.md) or feature specs. Bad ideas get archived with a reason.

## How This Works

1. **Capture immediately.** When an idea comes up (conversation, research, testing, feedback), add it here. Don't evaluate yet.
2. **Review regularly.** Every 2 weeks (or at the start of each feature cycle), review this list:
   - Promote strong ideas to DESIGN_PRINCIPLES.md (if they're principles) or to a feature spec (if they're features)
   - Archive rejected ideas to the bottom with a reason (never delete -- the reason is valuable)
   - Leave undecided ideas for next review
3. **Tag with source.** Where did this idea come from? Research, user feedback, competitor analysis, design session, implementation insight?
4. **Check implementation.** When completing a feature, scan this list: did we use the ideas we meant to? Did we miss any that apply?

### Handbook Errata + Discovery (apply-errata-and-afh-mosaic WP)

These ideas surfaced during the apply-errata-and-afh-mosaic build on 2026-04-28. Status: all `New` until reviewed.

- **PHAK MOSAIC bullet-edits parser archetype.** The PHAK October 2025 MOSAIC addendum uses a different layout than AFH MOSAIC: comma-delimited chapter anchors ("Chapter 1, Introduction to Flying, X section, ...") and bullet-level edits ("in the X bullets on page Y-Z, the bullet that states ... will change to state ..."). Needs a new `bullet-edits` parser archetype before PHAK MOSAIC can apply. AFH MOSAIC ships via the existing additive-paragraph parser; PHAK YAML carries the erratum entry but apply-time raises `UnknownErrataLayoutError` until the new parser exists.
- **Hangar UI for errata discovery + apply orchestration.** Each new CLI command (`--apply-errata`, `--reapply-errata`, `discover-errata`) is shaped to be wrappable. When `apps/hangar/` reawakens (per the post-FIRC-migration roadmap), build a thin SvelteKit surface that calls these via the existing dispatcher or HTTP. Help text on each CLI flag carries a "Future: hangar UI wraps this command" note so the wrapper-target stays discoverable. PRD lives on the hangar app once it exists.
- **Section-level "what changed since I last read this?" notification.** Once errata are applied, the reader has the data to surface "this section was amended on `applied_at` after you last read it" in the dashboard. UI is future work; the data shape (`handbook_section_errata.applied_at` joined against `handbook_read_state.updated_at`) supports it directly.
- **Auto-apply errata once parser-coverage gate is met.** Spec lists the gate as "≥3 distinct addendum layouts proven via unit tests + dry-run zero-diff against human-applied output." Today: 1 layout (additive-paragraph) shipped. Trigger: when `bullet-edits` and a third archetype land, flip the auto-apply switch and remove the manual review step. Each layout costs roughly half a phase to build; cadence depends on FAA publication.
- **Onboard W&B / WSC / PPC handbooks (signal-only today).** The October 2025 MOSAIC release also amended the Weight & Balance Handbook, the Weight-Shift Control Handbook, and the Powered Parachute Handbook. None of these are ingested today; their MOSAIC addenda will appear as `signal-only` candidates in the discovery scan (R7) until the handbooks themselves are ingested. Trigger: when a content priority justifies one of these books, ingest it then apply its addenda.
