# Spec: Theme picker -- shared library and multi-app rollout

Extract the picker shipped in #183 into a shared `@ab/themes/picker/*` surface. Re-point study at the shared API. Wire the same surface into sim and hangar so adding the picker to a new app is roughly five lines of glue.

## Why

Phase 9 (#183) shipped the picker in study only. The cookie name, parser, override rule, pre-hydration script, and validation endpoint were a single copy then -- but the override rule (`/sim/*` route lock + `sim/glass` forces dark) is safety-critical, and the pre-hydration `app.html` script can't `import` from the bundle. Shipping that bundle to sim and hangar via copy/paste would create three drift sites for "must stay in sync with `listThemes()`."

## Pre-hydration approach: Option A (build-time codegen)

Each app's `app.html` substitutes a `%theme-pre-hydration%` placeholder:

```html
<script>%theme-pre-hydration%</script>
```

`hooks.server.ts` does the substitution via `resolve(event, { transformPageChunk })`, importing the script body from `@ab/themes/generated/pre-hydration`. The same generated module exports `PRE_HYDRATION_SCRIPT_CSP_HASH`, which `svelte.config.js` imports into its `script-src` directive so a strict CSP allows the inline script to run.

Why Option A over a runtime function (Option B):

- The CSP hash must be a literal at config-load time. SvelteKit's CSP config is synchronous; awaiting a Promise-shaped hash function from inside `svelte.config.js` is a non-starter.
- The codegen artifact is committed, so CI and prod see byte-identical output. Runtime computation would cost a hash on first request per process and reintroduce a "scripts/themes/emit" determinism check.
- A Vite plugin would have worked but adds a per-app config dependency. The codegen path is one script, called from `bun themes:emit`, that already runs in the same step that writes `tokens.css`.

## Override rule (unchanged from #183)

1. Route safety lock wins (`/sim/*` -> `sim/glass`).
2. User pref wins on every other route.
3. `sim/glass` forces dark appearance regardless of how it was reached.
4. No user pref set -> path default (`resolveThemeForPath`).

## Out of scope

- Migrating login pages to the picker. They run outside the (app) layout and pin a theme deliberately (study -> sectional, hangar -> default). Drop.
- A11y announcement on theme change. Inherited from the Phase 9 deferral; trigger unchanged.
